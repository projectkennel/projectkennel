# AS-BUILT.md — documentation ↔ code reconciliation

> Generated 2026-06-04 by a 22-area multi-agent audit (one auditor per authoritative doc vs its code, then an adversarial verifier per area). Read-only — no source was changed. The verifier dropped false positives; the four highest-stakes security findings were additionally re-verified by hand (✔︎ noted inline). Treat `file:line` refs as of HEAD `0302c35`. Spot-check before acting — this is a map, not a court verdict.

## Legend

- 🔴 **CONTRADICTS** — code does the opposite of the doc.
- ⚪ **DOC-ONLY** — documented, but no implementation exists (aspirational).
- 🟠 **DRIFTED** — built, but diverged (different path / name / value / signature).
- 🟡 **PARTIAL** — partly built; the doc oversells completeness.

## Top line

**151 confirmed divergences** across 22 areas — 9 contradicts, 57 doc-only, 57 drifted, 28 partial. By severity: 40 high, 70 medium, 41 low.

They fall into three directions — important, because only one is a regression:

1. **Docs ahead of code (aspirational, never built).** Whole subsystems and controls are written up as if they exist: D-Bus proxy, X11 isolation, `fs.scrub`/`fs.home.sanitise`, runtime TTL enforcement, SO_PEERCRED, the single-instance flock, bind-port policy, the `.replace` delta operator, auto-compile-on-run. **And — the load-bearing one — the `exec.allow` allowlist is not an execution control.**
2. **Code ahead of docs (built, docs stale).** `kennel-audit` is a real first-class crate (02-6/03/06 still say it doesn't exist); `kennel-ssh-reorigin` and `kennel-socks-connect` exist; the real spawn API is `Plan`+`prepare`/`spawn` (not the documented `Spawn`/`Workload`); the privhelper op set is add/del-addr + setup-egress + set-gid-map (the documented cgroup-create/delete ops were retired — kenneld now makes the cgroup unprivileged).
3. **Genuine drift (built differently than documented).** `/opt/kennel` vs the documented `/usr/libexec/kennel`; the socket is `control.sock` not `kenneld.sock`; the daemon trust dir is `/etc/kennel/trust` while everything else uses `/etc/kennel/keys`; netproxy listens on TCP loopback, not the documented `proxy.sock` unix socket; `ttl_action` is `stop`/`warn` not the documented `exit`/`warn`/`renew`.

The single most important takeaway: **08-as-built-notes.md, 02-6-internal-api.md, and the whole `02-*` spec layer are the least trustworthy docs** — they describe an earlier architecture. The `design/07-*` subsystem docs describe an intended feature set that is only partly built. Trust the code; use this map to know which docs lie.

## Triage log (resolution direction — code vs doc)

This audit reports *what* diverged, not *which side is wrong*. A finding's `status` (drifted/contradicts/…) is NOT a verdict that the code is at fault — in several cases reviewed with the maintainer the **code is authoritative and the doc is stale**. Confirmed rulings so far:

- **IPv4/IPv6 loopback addressing** (01-process-model finding) — **RESOLVED, doc-side.** The code's bit-packed `/28` (`127 | tag(12) | ctx(8) | host(4)`) was correct; the doc was wrong and was fixed in `f9bbf47` *"docs: correct the per-kennel loopback addressing to match the code."* No code change.
- **privhelper cgroup-create/delete ops (bytes 3 & 4)** (01-process-model + 04-trust-boundaries "boundary 1" findings) — **DOC-FIX needed, code is right.** The ops were intentionally dropped: the cgroup is created higher up in kenneld's delegated subtree, and a privhelper-creates-the-cgroup design can't work given that structure. The docs describing those ops / that boundary are stale.
- **`InstallConstants` findings** (02-2 / 02-6 / 09 / paths) — **STALE.** The type was removed in `2db5eff refactor(policy): defer <tag>/<gid> to the daemon; drop InstallConstants`; any finding citing `InstallConstants` no longer applies.
- **Hardcoded `/opt/kennel` paths + trust-store split-brain** (H35/H37/H38, D4) — **FIXED (config layer, D11).** New `kennel-config` crate resolves deployment paths from a root-owned `system.toml` cascade (`/usr/lib/kennel` → `/etc/kennel`, default `libexec_dir=/usr/libexec/kennel`, `trust_dir=/etc/kennel/keys`); all `/opt` constants deleted, daemon + CLI wired, vendor config shipped, `install.sh` moved to `/usr/libexec/kennel` and now installs the AppArmor profile (G2) and all binaries (G5). The daemon's `/etc/kennel/trust` constant (H37's bug) is gone.

## ⚠️ Security-relevant: documented controls that are NOT enforced as described

These are the ones that matter for the threat model — a reader trusting the doc would believe a control is active that is not. All four hand-verified.

### 1. `exec.allow` is not an execution allowlist  ✔︎ verified
The design (`design/07-1-exec.md`) says execution is gated by Landlock `EXECUTE` on the `exec.allow` set. In reality `exec.allow` is read **only** to validate that placeholders were substituted ([kennel-spawn/src/lib.rs:164](src/crates/kennel-spawn/src/lib.rs#L164)); it never becomes a Landlock rule or any runtime check. Execution is governed entirely by `fs.read`/`fs.write`, whose grant includes `AccessFs::EXECUTE` ([plan.rs:136](src/crates/kennel-spawn/src/plan.rs#L136), populated at [plan.rs:362-385](src/crates/kennel-spawn/src/plan.rs#L362)). **Consequence:** anything on a readable path is executable. The `inspect-only` template grants `fs.read = /usr/**` while listing only cat/grep/… in `exec.allow` — so its stated guarantee ("nothing in the kennel can execute the repo's code") does **not** hold; all of `/usr/bin` is runnable. `exec.deny` is likewise parsed but never composed up the chain.

### 2. `deny_setuid`/`deny_setgid`/`deny_setcap` flags are never read  ✔︎ verified (mitigated)
The three flags are asserted as mandatory-true invariants and carried into the settled policy, but no spawn-path code consumes them. **However**, the *effect* is largely achieved structurally: every constructed mount is `MS_NOSUID` ([kennel-syscall/src/mount.rs:88-129](src/crates/kennel-syscall/src/mount.rs#L88)) and `PR_SET_NO_NEW_PRIVS` is set, so setuid/setgid execution is blocked regardless of the flag. So this is "enforced by a different mechanism than documented," not an open hole — but the per-file-capability ("setcap bit") case is not separately checked.

### 3. No SO_PEERCRED check on the control socket  ✔︎ verified
`02-4-ipc.md` and trust boundary 7 (`04-trust-boundaries.md`) state kenneld verifies the connecting peer's UID via SO_PEERCRED and rejects mismatches. No such check exists anywhere in kenneld or the privhelper (grep: none). The `serve()` accept loop services every connection. The actual protection is the socket's `0600` mode under `$XDG_RUNTIME_DIR` (same-uid only), so this is defence-in-depth that's missing, not a wide-open door — but the documented boundary is unbuilt.

### 4. No single-instance flock  ✔︎ verified
`05-state-and-supervision.md` describes a per-user `kenneld.lock` flock and EADDRINUSE fail-on-second-instance. Neither exists: `socket::bind()` **removes** any existing socket first ([socket.rs:52](src/crates/kenneld/src/socket.rs#L52)), so a second daemon silently takes over the socket rather than failing closed.

## Per-area reconciliation

### docs/architecture/01-process-model.md vs kennel-spawn / kennel-syscall namespace

**Built & working:** The doc's core userns/seal/fork/PID-1 model is faithfully built. The production spawn path (plan default Namespaces::USER|MOUNT|PID|IPC) establishes an identity-mapped user namespace unprivileged (establish_identity_userns: deny setgroups, identity uid_map + gid_map), gains CAP_SYS_ADMIN inside it, unshares mount/IPC/PID, then double-forks via fork_into_pid1 so the workload is PID 1 of the new PID namespace and can mount a fresh /proc; the intermediate reaper A closes fds>=3, waits on B, and exits with B's status. The seal (cgroup-join in the outer seal; pivot_root view, no_new_privs unconditionally, seccomp, Landlock in the inner seal) runs post-fork pre-execve and fails closed. The §7.2.8 deferred-gid_map handshake (establish_userns_defer_gid_map + send_ready/recv_ack servicer thread) is also built. The divergences are in the doc's privhelper op list and its network-address/cgroup-path notation, not the spawn mechanism.

- 🔴 CONTRADICTS · **HIGH** — kennel-privhelper has a 'Create a cgroup under /sys/fs/cgroup/kennel/<kennel>/' operation, and kenneld invokes the privhelper for 'cgroup creation' during bring-up and 'deletes the cgroup' on teardown.  
  *doc:* `docs/architecture/01-process-model.md:41,28,181,183` → *reality:* The privhelper's cgroup create/delete ops (op bytes 3 and 4) are retired; the wire enum exposes only AddAddr(1), DelAddr(2), SetupEgress(5), set-gid-map(6). - 🟠 DRIFTED · **HIGH** — The privhelper's operation set is: add v4 loopback addr, add v6 ULA addr, remove addrs, create cgroup. The IPC diagram and op list describe exactly these.  
  *doc:* `docs/architecture/01-process-model.md:38-41,161` → *reality:* Actual ops are add-addr, del-addr, setup-egress (load/populate/attach the egress BPF programs to a kennel's cgroup), and set-gid-map (the §7.2.8 privileged multi-gid map write).   *doc:* `docs/architecture/01-process-model.md:41` → *reality:* The plan builds the cgroup path as /sys/fs/cgroup/<namespace>/<ctx> where <namespace> is the caller's resource namespace from /etc/kennel/subkennel and <ctx> is the numeric context byte, not the kennel name.   *doc:* `docs/architecture/01-process-model.md:43,51,146,169` → *reality:* IPv4 is bit-packed 127 | tag(12 bits) | ctx(8 bits) | host(4 bits): e.g. tag 9, ctx 5, host 1 yields 127.0.144.81 (a /28, 16 host addrs), NOT 127.9.5.1 in a /24 — the doc's dotted '127.<tag>.<ctx>.1' notation does not produce the real proxy address and the prefix length is wrong.   *doc:* `docs/architecture/01-process-model.md:118,26,182` → *reality:* On the production userns path the workload (B) is PID 1 of a new PID namespace whose parent is the intermediate reaper A, not kenneld. kenneld's Child handle is for A; A reaps B and _exit()s with B's status, relaying the exit code. ### docs/architecture/02-5-bpf-abi.md

**Built & working:** The map and event ABI is genuinely well-aligned: all 8 documented programs (connect4/6, bind4/6, setsockopt, sock_create, sendmsg4/6) exist with the documented cgroup attach types in the loader's KENNEL_PROGRAMS table, and the C structs in bpf/maps.h and bpf/audit_events.h match the documented layouts (kennel_meta 64B with magic/abi/ctx/proxy fields; allow_entry 8B; LPM v4/v6 keys 8/20B; bind_subnet 28B; audit_hdr + connect/bind/sock/sockopt payloads; magic/version/kind enum values). The Rust loader mirrors these sizes by hand in KENNEL_MAPS and loads/relocates/attaches without CO-RE; root-tests prove connect4 fails-closed, allows on a populated map, and drains an AUDIT_NET_CONNECT_DENY event. The deny-first lookup, fail-closed semantics, and ringbuf drain are real.
