# API surfaces — CLI

## Stability commitment

**Stable** per `02-0-overview.md`. Subcommands, flags, exit codes, and machine-readable output (`--output=json` and `--output=jsonl`) are backwards-compatible across minor versions. Human-readable text output is best-effort and may evolve for clarity; scripts must use a machine-readable output mode rather than parsing the human format.

Deprecations follow the policy in `02-0-overview.md` §Deprecation policy: announced in CHANGELOG, warned at runtime, minimum one minor version of overlap before removal.

The CLI's version is reported by `kennel --version` as a SemVer string. The project's version and the CLI's version are the same.

---

## Invocation

```
kennel [GLOBAL FLAGS] <SUBCOMMAND> [SUBCOMMAND FLAGS] [ARGUMENTS]
```

The CLI is the user's entry point. It is stateless: every invocation starts from disk, queries kenneld for shared state if available, and proceeds independently. `kennel` does not maintain a long-running connection across invocations.

### Global flags

| Flag | Argument | Effect |
|---|---|---|
| `-h`, `--help` | — | Print help for the CLI or the named subcommand and exit. |
| `-V`, `--version` | — | Print SemVer string and exit. |
| `-v`, `--verbose` | — | Increase log verbosity to stderr. Repeatable: `-vv` for trace. Does not affect audit log. |
| `-q`, `--quiet` | — | Suppress non-error stderr output. |
| `--config-dir` | `<path>` | Override `~/.config/kennel/` for this invocation. |
| `--state-dir` | `<path>` | Override `~/.local/state/kennel/` for this invocation. |
| `--output` | `human` \| `json` \| `jsonl` | Output format. Default `human`. Subcommands documented as producing structured output respect this; subcommands that are purely interactive (`run`) ignore it for the workload's own stdout/stderr but use it for their own status output. |
| `--no-kenneld` | — | Force degraded mode: do not contact kenneld, invoke privhelper directly, spawn daemons as CLI children. Used for one-shot kennels and for debugging. |
| `--kenneld-socket` | `<path>` | Override `/run/user/<uid>/kennel/kenneld.sock`. Used for testing. |

Global flags must appear before the subcommand. Subcommand-specific flags appear after the subcommand name.

## Subcommands

### `kennel run` — start a workload inside a kennel

```
kennel run <kennel> [--policy <path>] [--template <name>] [--detach] [--name <id>] [--no-compile] -- <command> [args...]
```

Spawns a workload inside the named kennel. If the kennel is not running, it is started; if it is running, the new workload joins (sharing the same cgroup, same daemons, same network allowlist). The `--` separator delimits CLI flags from the workload command and its arguments.

Arguments and flags:

- `<kennel>` — the kennel name, resolved against `~/.config/kennel/kennels/<kennel>.toml`.
- `--policy <path>` — override: use this policy file rather than the named kennel's default. The kennel name is still used as the kennel ID for daemon sharing.
- `--template <name>` — start an ad-hoc kennel using a named template with no leaf policy. The kennel is named with a generated ID; the kennel exits when this `kennel run` exits regardless of grace period.
- `--detach` — fork into background after the workload reports ready. The CLI returns 0 immediately; the kennel survives. Audit events for the lifecycle continue to be written.
- `--name <id>` — give the spawned kennel a specific name (for use with `--template`).
- `--no-compile` — require an existing settled policy; fail rather than auto-compiling a source policy. The default in attested deployments.

Exit code is the workload's exit code (passed through). If the kennel could not be started, exit codes from the *kennel startup* range apply (see below) and the workload is not executed.

stdout/stderr of the workload are inherited from the CLI's stdout/stderr by default. The CLI does not interpose; what the workload prints appears in the user's terminal exactly as if it had been invoked directly.

Signal handling: `SIGINT`/`SIGTERM` to the CLI are forwarded to the workload's process group. The CLI does not exit until the workload has exited (or `--detach` was passed).

Source vs settled policy: `kennel run` enforces a *settled* policy (`02-2-config-schema.md` §The settled policy), never a source policy directly. If a fresh signed settled policy exists for the kennel, it is used as-is. In development mode, if the named policy is a source policy with no fresh settled artefact, `kennel run` auto-compiles it in memory (resolving the chain, verifying source signatures, checking the lockfile) and runs the result, marked a dev build. In attested deployments the workstation holds only signed settled policies and `kennel run` verifies the signature and spawns; no source resolution occurs. `--no-compile` forces use of an existing settled artefact and fails rather than auto-compiling.

### `kennel stop` — request a graceful stop

```
kennel stop <kennel> [--timeout <duration>]
```

Sends `SIGTERM` to all workloads in the named kennel. Waits up to `--timeout` (default `30s`) for them to exit. If they do not exit, returns non-zero; the caller may follow up with `kennel kill`.

Stopping a kennel does not immediately tear down its daemons; the grace period (`05-state-and-supervision.md`) applies. To bypass the grace period, use `kennel stop <kennel> --reap`.

Flags:

- `--timeout <duration>` — `5s`, `1m`, etc. Default `30s`.
- `--reap` — bypass the grace period; tear down daemons immediately after the last workload exits.

### `kennel kill` — force-kill a kennel

```
kennel kill <kennel>
```

Sends `SIGKILL` to all workloads in the kennel. Cannot be cleanly recovered from by the workload. Followed automatically by daemon reap (no grace period).

### `kennel list` — list known and running kennels

```
kennel list [--running] [--all]
```

Lists kennels. Default: all kennels defined in `~/.config/kennel/kennels/` *and* all running kennels.

Flags:

- `--running` — only kennels with at least one live workload.
- `--all` — include defined-but-never-started kennels.

`--output=json` produces an array of objects with `name`, `state`, `policy_path`, `workload_count`, `started_at` (or `null` if not running), `template_chain` fields.

### `kennel status` — detailed status for one kennel

```
kennel status <kennel>
```

Reports for the named kennel: state (`absent` / `starting` / `running` / `draining` / `stopped`), workload count, started-at, template chain (resolved), daemon PIDs, reference counter, drain timer (if `draining`), audit-log file path.

`--output=json` produces a single object matching the list-element shape with additional fields (`daemons`, `drain_remaining_seconds` if draining, `recent_deny_count`).

### `kennel audit` — query the audit log

```
kennel audit <kennel> [--since <duration>] [--until <duration>] [--resource <class>]
              [--event <name>] [--follow] [--limit <n>]
              [--print-journalctl-command]
```

Reads from the kennel's audit stream. Sources read depends on which sinks the kennel's `[audit]` policy enables (`02-3-audit-schema.md`):

- If the **file sink** is enabled, the CLI reads directly from the per-kennel JSONL files. This is the standard case.
- If only the **journald sink** is enabled, the CLI cannot read the journal directly — it does not link a journald client by default. Pass `--print-journalctl-command` to emit an equivalent `journalctl` invocation; pipe it to a shell to execute.
- If the **stdout sink** is the only one in use (typical container deployment), the events are in the orchestrator's log system; `kennel audit` reports that no local source is available and exits with code 4.

Default `--output` is `human` (one event per line with timestamps and key fields); `--output=jsonl` emits the audit events verbatim, suitable for piping to other tools.

Filter flags:

- `--since <duration>` / `--until <duration>` — relative time bounds (`1h`, `30m`, `2d`); `--since` defaults to `1h`, `--until` defaults to now.
- `--resource <class>` — `net` / `fs` / `exec` / `unix` / `dbus` / `priv` / `lifecycle`.
- `--event <name>` — exact event name (`net.connect-deny`, `fs.access-deny`, etc.).
- `--limit <n>` — cap on the number of events returned. Default 1000; explicit `--limit 0` means no cap.
- `--follow` — stream new events as they arrive; runs until SIGINT. Available for the file sink only.
- `--print-journalctl-command` — emit the equivalent `journalctl` invocation for the requested filters and exit. Useful when journald is the active sink.

The audit event schema and sink mappings are in `02-3-audit-schema.md`.

### `kennel validate` — validate a policy without running

```
kennel validate <policy-path> [--template-dir <path>] [--strict-invariants]
```

Loads the named policy, resolves its template chain, verifies signatures, applies invariant checks. Exits 0 on success; non-zero on any failure with a structured error message naming the field that failed.

Flags:

- `--template-dir <path>` — additional template search path.
- `--strict-invariants` — fail even on warnings (e.g., a delta that weakens a non-invariant rule with no `reason` field).

`--output=json` produces a `{"valid": bool, "errors": [...], "warnings": [...]}` object.

### `kennel compile` — resolve a source policy into a signed settled policy

```
kennel compile <policy> [--output-path <path>] [--key <key-path>] [--unsigned] [--template-dir <path>]
```

Resolves a source policy fully — inheritance chain, includes, deltas, source-signature verification, lockfile byte-checks, invariant and threat-tag validation, installation-constant substitution — and emits a **settled policy**: a flat, signed, canonical-TOML artefact the runtime consumes directly. See `02-2-config-schema.md` §The settled policy and design doc §9.10.

This is the step that moves resolution off the spawn hot path. In fleet deployments it runs centrally in CI; the signed settled policies are pushed to workstations that need not hold the templates or the lockfile.

Flags:

- `--output-path <path>` — where to write the settled policy. Default: `<policy-dir>/<name>.settled.toml`.
- `--key <key-path>` — the signing key for the settled policy. Required unless `--unsigned`.
- `--unsigned` — emit a content-sealed but unsigned settled policy (development use; the runtime accepts unsigned settled policies only in development mode, never in attested mode).
- `--template-dir <path>` — additional template search path for resolution.

Exit codes follow the validation range: 3 for policy-validation failure, 6 for a source signature or lockfile-pin failure encountered during resolution.

`--output=json` produces a `{"settled_path": ..., "content_sha256": ..., "resolved_artifacts": [...], "deferred_substitutions": [...]}` summary object.

### `kennel diff` — diff two policies

```
kennel diff <a> <b> [--threat-impact]
```

Computes the structural diff of two policies (or a policy versus the template it derives from). Emits a human-readable diff or, with `--output=json`, a structured diff object describing additions, removals, and modifications by section.

Flags:

- `--threat-impact` — annotate each diff hunk with the threat IDs the change exposes or addresses, drawn from `THREATS.md` and the policies' own threat tags.

### `kennel upgrade` — upgrade a kennel's template version

```
kennel upgrade <kennel> [--target-version <v>] [--dry-run]
```

Shows what would change if the kennel's pinned `template_version` were updated to the latest available (or the version named by `--target-version`). With `--dry-run` (default), only reports; without, applies the upgrade by rewriting the leaf policy's `template_version` field after the user confirms.

The diff is annotated with threat-impact tags the way `kennel diff --threat-impact` does.

### `kennel templates` — interact with available templates

```
kennel templates list [--installed-only]
kennel templates show <name> [--version <v>]
kennel templates inspect <name> [--version <v>]
kennel templates verify <name> [--version <v>]
```

- `list` — enumerate installed templates.
- `show` — print a template's resolved policy in canonical TOML form.
- `inspect` — print metadata (version, maintainer key fingerprint, threat tags, last-modified, declared invariants).
- `verify` — re-verify the template's signature against the configured signing-key set.

### `kennel hash` — print a canonical hash of a policy

```
kennel hash <policy>
```

Emits the SHA-256 of the policy's canonical-form serialisation. Used for build pipelines that want to detect policy drift, and by `kennel sign` (next).

### `kennel sign` — sign a template or policy

```
kennel sign <path> --key <key-path> [--detached]
```

For users with template-signing authority. Emits a signed envelope around the policy. `--detached` writes the signature to a side file rather than wrapping; the policy file remains as-is.

### `kennel completion` — generate shell completions

```
kennel completion bash | zsh | fish | elvish | powershell
```

Emits a shell-completion script to stdout. Stable surface (the completion script depends on the subcommand and flag set, which are stable).

### `kennel version`

```
kennel version [--full]
```

Same as `kennel --version`, with `--full` adding build metadata (commit hash, build date, supported MSRV, kernel-feature matrix the binary requires).

## Exit codes

| Code | Meaning |
|---|---|
| 0 | Success. For `kennel run`, the workload's own exit code is passed through. |
| 1 | Generic failure. Reserved for cases not covered by a more specific code. |
| 2 | Usage error: malformed command line, unknown subcommand, conflicting flags. |
| 3 | Policy validation failure: schema, invariants, or template inheritance. |
| 4 | Kennel not found: a named kennel does not exist in the config directory. |
| 5 | Permission denied: required kernel feature missing, privhelper unavailable, file permissions insufficient. |
| 6 | Signature verification failure: a template's signature did not verify. |
| 7 | Timeout: an operation (privhelper response, daemon readiness, `kennel stop --timeout`) timed out. |
| 8 | Concurrent-state conflict: e.g., kennel in `starting` and timed out waiting; rare. |
| 64–78 | Reserved for `sysexits.h`-style codes; not currently used. |

`kennel run` is special: when the workload runs, its exit code is returned, not one of the CLI's codes. To distinguish a kennel-startup failure from a workload exit, callers may pass `--output=json` to receive a structured result *and* check the exit code: workload exits return 0/$workload_code with `phase=workload` in JSON; startup failures return a CLI code with `phase=startup`.

## Output formats

`--output=human` (default) is a stable target for *display*, not for parsing. Whitespace, column widths, error wording, and the choice of which fields to include in `kennel list`'s default view may change in minor versions.

`--output=json` produces a single JSON document on stdout, ending with a newline, no trailing data. Suitable for `kennel list`, `kennel status`, `kennel validate`, `kennel diff`, `kennel templates show/inspect/verify`, `kennel version --full`.

`--output=jsonl` produces one JSON object per line, no surrounding container. Suitable for `kennel audit` and `kennel list --follow`-style streaming outputs.

JSON shapes are documented in the subcommand sections above. Top-level fields of stable shapes are *additive* across minor versions: new fields may appear, existing fields do not change name or type.

## Stability tracking

This chapter is the authoritative list of subcommands, flags, and exit codes. Adding a subcommand or flag is a minor-version event; removing one or changing semantics is a major-version event preceded by deprecation per `02-0-overview.md`.

CHANGELOG entries that touch the CLI go under `### CLI changes` and name the specific subcommand or flag affected.
