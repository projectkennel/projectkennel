# API surfaces — IPC wire formats

> **As-built status (see `08-as-built-notes.md` §8.1/§8.2).** The control protocol
> is implemented in `kenneld::control` (`Request`/`Response` + `read_frame`); the
> privhelper protocol in `kennel-privhelper::wire`. There is no `kennel-ipc*` crate
> (the §357 cross-reference is stale). Two corrections to this chapter: the length
> prefix is **native-endian** (not big-endian — both ends are the same host), and
> the **version handshake below is not yet built** (the control socket goes
> straight to request/response). The privhelper request is **294 bytes**, not 292.
> The framing/encoding rules are otherwise as-built.

## Stability commitment

**Internal-stable** per `02-0-overview.md`. The IPC protocols described here may change between minor versions, but Project Kennel ensures internal consistency within a release: the CLI, kenneld, and privhelper in one release agree. External consumers do not write code against these surfaces; an operator's tooling that wants to query kenneld goes through the CLI.

Each protocol begins with a version handshake. Both sides reject and disconnect on mismatch — there is no inter-version compatibility shim. Within a release, both sides are built from the same source, so the handshake's purpose is to catch misinstall (CLI from package A talking to kenneld from package B) rather than to negotiate.

The wire format is documented here so that reviewers and auditors can verify the protocol's safety properties. It is not a contract to consumers.

---

## Common transport

All Project Kennel IPC uses Unix domain sockets, `SOCK_STREAM` type, owned by the user (mode 0600 unless otherwise noted). No TCP. No abstract-namespace sockets.

**Framing.** Each direction is a sequence of framed messages. Each frame:

```
+-----------------+-----------------+-----------------+
| length (u32 NE) | body (length B) | (next frame)    |
+-----------------+-----------------+-----------------+
```

- `length` is the byte length of `body`, **native-endian** unsigned 32-bit (a same-host transport; consistent with the native-endian encoding note below). `read_frame` checks `length > MAX_MESSAGE` (1 MiB) **before** allocating the body buffer.
- `body` is a fixed-layout binary message (a struct, keyed by a leading message-type byte). Length is capped at 1 MiB; longer frames are a protocol violation (sender error; recipient closes the connection).
- The recipient reads `length`, then reads exactly that many bytes, then decodes the body into the typed message for its type byte. Partial reads are stitched together by the recipient.

**Encoding.** Bodies are **classic struct messages** — a fixed byte layout per message type, not a serialisation language (no JSON/serde): the overhead of an abstract format is unnecessary for a same-machine, fixed-shape protocol. Multi-byte integers are native-endian; floats are not used. The privhelper protocol below is the implemented exemplar (`kennel-privhelper`'s `wire` module); the kenneld socket message structs are defined as those crates land. Decoding is field-by-field, so the IPC crates need no `unsafe`.

**Authentication.** Connecting clients are identified by `SO_PEERCRED`. The server (kenneld, privhelper) verifies that the client's UID matches the expected value and rejects otherwise. There is no other authentication; the socket file mode is the access-control mechanism.

## Handshake

The first frame in each direction is a version handshake.

Client → server:

```json
{
  "kind": "hello",
  "client": "kennel-cli",
  "client_version": "0.4.2",
  "protocol_version": 1
}
```

Server → client:

```json
{
  "kind": "hello-ack",
  "server": "kenneld",
  "server_version": "0.4.2",
  "protocol_version": 1,
  "session_id": "01HZX..."
}
```

If `protocol_version` differs:

```json
{
  "kind": "hello-ack",
  "error": {
    "code": "version-mismatch",
    "message": "kenneld supports protocol 1; client requested 2",
    "supported": [1]
  }
}
```

The server then closes the connection. The client surfaces a structured error to the user.

`session_id` is a ULID assigned by the server; it appears in audit events for correlation across the session.

## CLI ↔ kenneld control protocol

Socket: `/run/user/<uid>/kennel/kenneld.sock`, mode 0600, owned by the user.

After the handshake, the client issues one or more requests; the server responds to each in order. The protocol is request-response, with one additional capability: streaming subscriptions for audit-event follow.

### Request envelope

```json
{
  "kind": "request",
  "id": "1",
  "method": "kennel.start",
  "params": {
    "name": "ai-coding",
    "policy_path": "/home/u/.config/kennel/kennels/ai-coding.toml",
    "extra_workload": false
  }
}
```

- `id` is a client-chosen string, unique within the connection. The response echoes it.
- `method` is the operation name; see §Methods.
- `params` is the method-specific payload.

### Response envelope

```json
{
  "kind": "response",
  "id": "1",
  "result": { ... }
}
```

or, on error:

```json
{
  "kind": "response",
  "id": "1",
  "error": {
    "code": "policy-validation",
    "message": "invalid template_version 'v5': not found",
    "details": { ... }
  }
}
```

Error codes are stable strings within a `protocol_version`. The full catalogue:

| Code | Meaning |
|---|---|
| `not-found` | Named kennel does not exist. |
| `policy-validation` | Policy file failed schema or invariant checks. |
| `signature-failure` | Template signature did not verify. |
| `kernel-feature-missing` | A required kernel feature is unavailable. |
| `privhelper-unavailable` | The privhelper could not be invoked. |
| `privhelper-refused` | The privhelper rejected a request as out-of-scope. |
| `state-conflict` | Operation incompatible with current state (e.g., `stop` on `absent`). |
| `concurrent-timeout` | Waited for another in-flight operation; gave up. |
| `internal` | Server-side bug; details carry diagnostic info. |

### Streaming responses

For methods that stream (`audit.subscribe`), the server emits a sequence of frames:

```json
{ "kind": "stream", "id": "2", "item": { ...event... } }
{ "kind": "stream", "id": "2", "item": { ...event... } }
...
{ "kind": "stream-end", "id": "2" }
```

The client may cancel:

```json
{ "kind": "cancel", "id": "2" }
```

The server responds with `stream-end` once the cancel takes effect.

### Methods

**`kennel.start`** — start a kennel and prepare to accept workload spawn.

Params: `name` (string), `policy_path` (string, optional — defaults to the standard location for the named kennel), `template` (string, optional — for ad-hoc kennels).

Result:

```json
{
  "kennel_uuid": "01HZX...",
  "ctx": 5,
  "loopback_v4": "127.42.5.1",
  "loopback_v6": "fd42:cafe:5::1",
  "audit_dir": "/home/u/.local/state/kennel/ai-coding/",
  "proxy_socket": "/run/kennel/ai-coding/proxy.sock",
  "ssh_agent_socket": "/run/kennel/ai-coding/ssh-agent.sock",
  "cgroup_path": "/sys/fs/cgroup/kennel/ai-coding"
}
```

The CLI uses this to construct the spawn environment for the workload.

**`kennel.workload-attaching`** — the CLI signals it is about to fork the workload.

Params: `kennel_uuid` (string), `client_pid` (int).

Increments kenneld's reference counter for the kennel. Result is a simple acknowledgement.

**`kennel.workload-exited`** — the CLI signals the workload has exited.

Params: `kennel_uuid` (string), `client_pid` (int), `exit_code` (int or null), `signal` (string or null).

Decrements the reference counter. If the counter reaches zero, kenneld starts the drain timer.

**`kennel.stop`** — request a graceful stop.

Params: `name` (string), `timeout_seconds` (float, default 30), `reap_immediately` (bool, default false).

**`kennel.kill`** — force-kill.

Params: `name` (string).

**`kennel.list`** — list kennels.

Params: `include_stopped` (bool), `include_defined` (bool).

Result: `{ "kennels": [ ...status objects... ] }`.

**`kennel.status`** — detailed status for one kennel.

Params: `name` (string).

Result: the full status object.

**`audit.query`** — historical audit-log query.

Params: `name` (string), `since_seconds` (float or null), `until_seconds` (float or null), `resource` (string or null), `event` (string or null), `limit` (int, default 1000).

Result: `{ "events": [ ...audit event objects... ], "truncated": bool }`.

**`audit.subscribe`** — streaming audit-event follow.

Params: same as `audit.query`, plus `historical_first` (bool — include matching historical events before streaming live ones).

Response is a stream (see §Streaming responses).

**`policy.validate`** — validate without running.

Params: `policy_path` (string), `template_dir` (string or null), `strict_invariants` (bool).

Result: `{ "valid": bool, "errors": [...], "warnings": [...] }`.

**`kenneld.status`** — kenneld's own self-status.

No params.

Result: `{ "uptime_seconds": ..., "kennels_active": ..., "kennels_draining": ..., "build_version": ..., "kernel_features": [...] }`.

**`kenneld.reload`** — reload policy caches and rotate audit logs.

No params. Triggered also by `SIGHUP`.

## kenneld ↔ privhelper protocol

The privhelper is short-lived per invocation, as documented in `01-process-model.md`. Each invocation runs `exec()` on the privhelper binary, the caller writes one fixed-layout request struct to the helper's stdin, the helper writes one fixed-layout response struct to stdout, the helper exits.

The wire format is **classic struct messages** — a fixed byte layout, not a serialisation language (no JSON/serde). The overhead of an abstract serialisation format is unnecessary for a same-machine, fixed-shape protocol. The authoritative definition is `kennel-privhelper`'s `wire` module (`Request`, `Response`); this section summarises it.

No version handshake — the helper version is the binary on disk, which the caller can read at exec time if it wants. Compatibility is enforced by the build (kenneld and privhelper come from the same release).

### Request

A single fixed 294-byte request struct on stdin (native-endian; helper and caller are the same machine). The `ctx` field is a `u16` (a kennel context can exceed 255 for v6-only kennels), and the cgroup create/delete ops were retired (`523c74a`) — kenneld manages cgroups unprivileged, so the helper's surface is exactly add-addr / del-addr / setup-egress:

```text
0      op           u8     (1 add-addr, 2 del-addr, 5 setup-egress)
1      family       u8     (4 or 6; 0 for the egress op)
2      prefix       u8
3      _reserved    u8     (0)
4..6   ctx          u16    (16-bit kennel context; a v4-enabled kennel uses ctx <= 255)
6..22  addr         [u8;16] (v4 in the first 4 bytes)
22..38 interface    [u8;16] (NUL-padded; kernel IFNAMSIZ)
38..294 cgroup_path [u8;256] (NUL-padded; the egress op's target cgroup)
```

The setup-egress op carries a variable-length `EgressPayload` tail after the fixed header (the `kennel_meta` + the four length-prefixed LPM arrays), bounds-checked on decode. The helper does not trust the request for authorisation (the caller is kenneld); it validates every field against the reserved scope (the installation `tag` + ULA GID) and the cgroup's ownership before any privileged syscall.

### Response

A single fixed 6-byte response struct on stdout:

```text
0      status   u8     (0 ok, 1 refused, 2 protocol, 3 internal)
1      refusal  u8     (refusal code when status==1, else 0)
2..6   errno    i32    (OS errno when status==3, else 0)
```

The helper exits with code `0` for `ok`, `1` for `refused`, `2` for protocol errors (malformed request, unknown operation), `3` for internal errors — matching `status`.

### Operations

| Operation | Params | Notes |
|---|---|---|
| `addr-add` | `interface`, `addr`, `prefix` | Add an IPv4 or IPv6 address to the named interface. `prefix` is an integer mask length. Validates `addr` is within the per-kennel allocation. |
| `addr-remove` | `interface`, `addr`, `prefix` | Remove a previously-added address. |
| `cgroup-create` | `path` | Create a cgroup under `/sys/fs/cgroup/kennel/`. Validates `path` is under that prefix. |
| `cgroup-delete` | `path` | Remove an empty cgroup. Refuses if non-empty. |

The helper refuses any request whose parameters fall outside Project Kennel's reserved ranges (per `01-process-model.md` §kennel-privhelper). The refusal is structured: code `out-of-scope`, message naming the rejected value.

### Locking

The helper takes `flock` on `/run/kennel/privhelper.lock` (mode 0600, owned by root) before any privileged syscall. Concurrent invocations serialise; the lock is held for the duration of one operation.

## kenneld ↔ per-kennel daemons

Per-kennel daemons (netproxy, dbus-proxy, ssh-agent) have control sockets at `/run/kennel/<id>/<daemon>.ctl` (mode 0600). The control protocol is a minimal subset of the kenneld control protocol's frame format.

### Methods (netproxy)

**`config.load`** — load resolved policy fragment.

Params: the network section of the resolved policy.

Result: ack or error.

**`config.reload`** — re-read configuration after a policy change.

Params: as `config.load`.

**`drain`** — stop accepting new connections; let existing connections complete; exit when all are done.

No params.

**`status`** — runtime status.

No params.

Result: `{ "uptime_seconds": ..., "active_connections": ..., "bytes_in": ..., "bytes_out": ..., "denies_since_start": ... }`.

### Methods (dbus-proxy)

Project Kennel uses `xdg-dbus-proxy` as an external dependency. Its control surface is not Project Kennel's; the kenneld interaction with it is via process signals (`SIGTERM` for drain) and the xdg-dbus-proxy filter file (regenerated on policy reload).

### Methods (ssh-agent)

The per-kennel `kennel-ssh-agent` (or stock `ssh-agent`) speaks the standard SSH agent protocol on its socket. Project Kennel does not extend it. kenneld manages the lifecycle (spawn, signal, reap) but does not speak the protocol.

## Protocol invariants

For every protocol described here:

- The server validates every field of every request before acting. Unknown fields are rejected.
- String fields have explicit length caps documented in the schema; reads are bounded.
- The server never reflects untrusted strings back to the client without sanitisation (per CODING-STANDARDS.md §10.3).
- Numbers outside their declared range are rejected at parse time, not at semantics time.
- Errors are first-class responses, not connection drops. The server only drops the connection on:
  - Protocol-framing violation (length exceeds cap, body is not UTF-8 JSON).
  - Handshake mismatch.
  - Client identity check failure (`SO_PEERCRED` UID mismatch).
  - Idle timeout (the server may close idle connections after 5 minutes; the CLI reconnects).
