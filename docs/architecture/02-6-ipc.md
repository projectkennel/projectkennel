# API surfaces — IPC wire formats

The control protocol is implemented in its own crate `kennel-lib-control`
(`Request`/`Response` plus `read_frame`), shared by the daemon and the CLI and
re-exported as `kenneld::{control, socket}`; the privhelper protocol in
`kennel-privhelper::wire`.

## Stability commitment

**Internal-stable** per `02-0-overview.md`. The IPC protocols described here may change between minor versions, but Project Kennel ensures internal consistency within a release: the CLI, kenneld, and privhelper in one release agree. External consumers do not write code against these surfaces; an operator's tooling that wants to query kenneld goes through the CLI.

There is no version handshake. Both ends of each socket are built from the same source within a release, so a connection goes straight to request/response with no negotiation. Misinstall (a CLI from one package talking to a kenneld from another) is prevented by packaging, not by an on-the-wire version exchange.

The wire format is documented here so that reviewers and auditors can verify the protocol's safety properties. It is not a contract to consumers.

---

## Common transport

All Project Kennel IPC uses Unix domain sockets, `SOCK_STREAM` type, owned by the user (mode 0600 unless otherwise noted). No TCP. No abstract-namespace sockets.

**Framing.** Each direction is a sequence of framed messages. Each frame:

```
+-----------------+-----------------+-----------------+
| length (u32 NE) | body (length B) | (next frame)    |
+-----------------+-----------------+-----------------+
```

- `length` is the byte length of `body`, **native-endian** unsigned 32-bit (a same-host transport; consistent with the native-endian encoding note below). `read_frame` checks `length > MAX_MESSAGE` (1 MiB) **before** allocating the body buffer.
- `body` is a fixed-layout binary message (a struct, keyed by a leading message-type byte). Length is capped at 1 MiB; longer frames are a protocol violation (sender error; recipient closes the connection).
- The recipient reads `length`, then reads exactly that many bytes, then decodes the body into the typed message for its type byte. Partial reads are stitched together by the recipient.

**Encoding.** Bodies are **classic struct messages** — a fixed byte layout per message type, not a serialisation language (no JSON/serde): the overhead of an abstract format is unnecessary for a same-machine, fixed-shape protocol. Multi-byte integers are native-endian; floats are not used. The control message structs live in `kenneld::control` and the privhelper structs in `kennel-privhelper`'s `wire` module. Decoding is field-by-field over a bounds-checked cursor, so neither module needs `unsafe`.

**Authentication.** Connecting clients are identified by `SO_PEERCRED`. The server (kenneld, privhelper) verifies that the client's UID matches the expected value and rejects otherwise. There is no other authentication; the socket file mode is the access-control mechanism.

## CLI ↔ kenneld control protocol

Socket: `/run/user/<uid>/kennel/control.sock`, mode 0600, owned by the user.

The client connects and issues a request directly; the server responds. There is no handshake frame. The protocol is request-response. A `Start` connection stays open after the first response: the daemon sends `Started` once the workload is running, then a single `Exited` when it terminates.

The request and response bodies are fixed-layout struct messages (`Request` and `Response` in `kenneld::control`), each a leading op/tag byte followed by primitively-encoded fields. There is no JSON, no request `id`, and no method-name string: the op byte *is* the method selector. Multi-byte integers are native-endian; string and list fields are `u32`-length-prefixed (a string is capped at `MAX_STRING` = 64 KiB, a list at `MAX_COUNT` = 4096 elements). UTF-8 is validated on decode.

### Requests

| Op | Request | Fields |
|---|---|---|
| 1 | `Start` | `policy` (path to the signed settled-policy file), `kennel` (name), `argv` (command + args), `cwd` (working directory) |
| 2 | `Stop` | `kennel` (name) |
| 3 | `List` | — |
| 4 | `AuthorizedKeys` | `key` (an offered public key as `"<type> <base64>"`) — the SSH bastion's root-owned `AuthorizedKeysCommand` (`kennel-akc`) queries the daemon for the forced-command lines bound to the key (§7.10.7) |

The workload's stdio is **not** carried in the request body — fds travel as `SCM_RIGHTS` ancillary data alongside the `Start` frame, and the frame's `interactive` flag says which shape. A **non-interactive** run passes three fds (the CLI's stdin/stdout/stderr) and the workload is attached to them directly. An **interactive** run passes a single connected socket; the spawn seal allocates a controlling pty inside the kennel's own devpts and returns its master over that socket for the CLI to proxy (design §7.9.5a). The fd transfer is the server/syscall layer's concern; the control module is the bytes.

### Responses

| Tag | Response | Fields |
|---|---|---|
| 0 | `Started` | `ctx` (u16, the allocated context number), `pid` (u32, the workload pid) |
| 1 | `Stopped` | — |
| 2 | `Listing` | array of `{ kennel (name), ctx (u16), pid (u32), running (bool) }` |
| 3 | `Exited` | `code` (i32; the workload exit status, or `128 + signal` if killed) |
| 4 | `Error` | `message` (a human-readable reason string) |
| 5 | `AuthorizedKeys` | `lines` (one `restrict,pty,command=…` line per matching edge; empty when no edge matches, so the bastion refuses the key) — the answer to the `AuthorizedKeys` request |

Errors are a first-class `Error` response carrying a message string, not a connection drop and not a coded error catalogue. A malformed frame (unknown op/tag, a truncated, oversized, or non-UTF-8 field) is a `WireError` at decode and closes the connection.

## kenneld ↔ privhelper protocol

The privhelper is short-lived per invocation, as documented in `01-process-model.md`. Each invocation runs `exec()` on the privhelper binary, the caller writes one fixed-layout request struct to the helper's stdin, the helper writes one fixed-layout response struct to stdout, the helper exits.

The exception is the `ConstructKennel` op (the factory path): it carries file descriptors and a variable-length Plan, so it runs over a `SOCK_SEQPACKET` socketpair (kenneld ↔ privhelper) rather than stdin/stdout, using `SCM_RIGHTS` ancillary data for the fds. It is documented in its own section below; the stdin/stdout transport above covers only the teardown `del-addr` op (the address *add* and the egress-BPF *attach* are folded into `ConstructKennel`).

The wire format is **classic struct messages** — a fixed byte layout, not a serialisation language (no JSON/serde). The overhead of an abstract serialisation format is unnecessary for a same-machine, fixed-shape protocol. The authoritative definition is `kennel-privhelper`'s `wire` module (`Request`, `Response`); this section summarises it.

No version handshake — the helper version is the binary on disk, which the caller can read at exec time if it wants. Compatibility is enforced by the build (kenneld and privhelper come from the same release).

### Request

A single fixed 294-byte request struct on stdin (native-endian; helper and caller are the same machine). The `ctx` field is a `u16` (a kennel context can exceed 255 for v6-only kennels). The stdin/stdout helper surface is now exactly one op — the teardown `del-addr` (the address *add* and the egress *attach* fold into the construction op, and `set-gid-map` was retired with the legacy spawn):

```text
0      op           u8     (2 del-addr; the construction op rides its own socketpair, below)
1      family       u8     (4 or 6)
2      prefix       u8
3      _reserved    u8     (0)
4..6   ctx          u16    (16-bit kennel context; a v4-enabled kennel uses ctx <= 255)
6..22  addr         [u8;16] (v4 in the first 4 bytes)
22..38 interface    [u8;16] (NUL-padded; kernel IFNAMSIZ)
38..294 cgroup_path [u8;256] (retained in the fixed layout; unused by del-addr)
```

The helper does not trust the request for authorisation (the caller is kenneld); it validates every field against the reserved scope (the installation `tag` + ULA GID) before any privileged syscall. (The `EgressPayload` — `kennel_meta` + four length-prefixed LPM arrays — now rides the construction datagram, framed after the construction-half; it is still bounds-checked on decode, and the cgroup's ownership is re-checked before the attach.)

### Response

A single fixed 6-byte response struct on stdout:

```text
0      status   u8     (0 ok, 1 refused, 2 protocol, 3 internal)
1      refusal  u8     (refusal code when status==1, else 0)
2..6   errno    i32    (OS errno when status==3, else 0)
```

The helper exits with code `0` for `ok`, `1` for `refused`, `2` for protocol errors (malformed request, unknown operation), `3` for internal errors — matching `status`.

### Operations

| Op | Operation | Params | Notes |
|---|---|---|---|
| 2 | `del-addr` | `family`, `interface`, `addr`, `prefix`, `ctx` | Remove a per-kennel address on teardown. Validates `addr` is within the per-kennel allocation before the netlink delete. The **only** standalone stdin/stdout op. |

The address *add* and the egress-BPF *attach* (the former byte-1 `add-addr` and byte-5 `setup-egress` ops) are folded into `ConstructKennel`: the loopback addresses ride the construction-half and the `EgressPayload` rides the construction datagram's tail, so a kennel is provisioned in **one** privileged spawn rather than four. The byte-6 `set-gid-map` op was retired with the legacy in-process spawn (the factory writes the full identity map, granted gids included, in one shot). The fixed `Request` byte values are left as-is so the wire encoding is stable.

Cgroup create/delete are not privhelper operations — kenneld manages cgroups unprivileged. The helper refuses any request whose parameters fall outside Project Kennel's reserved ranges (per `01-process-model.md` §kennel-privhelper). The refusal is structured: code `out-of-scope`, message naming the rejected value.

### The `ConstructKennel` factory op

The privhelper is the **factory** that builds the kennel (design §7.2): rather than kenneld assembling the sandbox unprivileged, kenneld asks the privhelper to construct it. Because this op carries file descriptors and a variable-length Plan, it does **not** use the stdin/stdout transport above — kenneld and the privhelper communicate over a `SOCK_SEQPACKET` socketpair, with `SCM_RIGHTS` ancillary data for the fds.

**Request.** A single `SOCK_SEQPACKET` datagram from kenneld:

- **Ancillary (`SCM_RIGHTS`):** for an interactive run, the pty return socket fd. The `kennel-bin-init` binary is **not** passed over the wire: the privhelper resolves it from the root-owned deployment config (`Deployment::kennel_bin_init()`), opens it itself pre-`clone`, and verifies root-ownership + non-writability before `fexecve` — so a compromised operator cannot choose what runs as the kennel's uid 0 (design §7.2.1, [`04-trust-boundaries.md`](../design/04-trust-boundaries.md)). It still `fexecve`s by descriptor (the host path is gone after `pivot_root`), just one it opened rather than received.
- **Body:** a length-prefixed frame, `[u32 ch_len][construction-half][egress payload]`. The **construction half** is the `kennel-lib-spawn::wire`-encoded uid/gid maps (`0 0 1` host-root + operator identity line + granted groups), the loopback config, the **per-kennel loopback addresses** to add (+ `ctx`, so the helper re-validates each against the reserved subnet), the binderfs params, the view bind list, and the pivot target. The length prefix lets the helper hand its decoder exactly the construction-half bytes; the remaining tail is the **`EgressPayload`** (the factory attaches the egress BPF before cloning). This half is parsed **host-side**, where there is no sandbox to manipulate it. The **supervision half** is *not* sent here — `kennel-bin-init` pulls it over binder from kenneld (`GET_SANDBOX_PLAN` → node 0; see [`02-4-binder.md`](02-4-binder.md) and design §7.2.3). The decoder is bounded and fuzzed.

The privhelper `clone`s the namespaces as the operator (so the user namespace is operator-owned); its child self-escalates to the kennel's uid 0 to build the root-owned surfaces (view, `/dev`, libraries, binderfs), mounts binderfs and chowns the device to the operator, `pivot_root`s, and `fexecve`s `kennel-bin-init` (PID 1) with empty argv/envp. The privhelper file caps are `cap_setuid` + `cap_setgid` + `cap_setfcap` (+ `cap_sys_admin` + `cap_net_admin`); the `0 0 1` map needs `cap_setuid`. The parent escalates to real root **only** to write the child's identity maps, then drops straight back to the operator and **exits as soon as it has reported the init pid** — it is not a reaper proxy. `kennel-bin-init` is PID 1 of the new namespace and outlives it; kenneld (a `set_child_subreaper`) adopts the orphaned init and `waitpid`s it for the workload's exit status. So there is no long-lived host-side factory process per kennel, privileged or otherwise. The factory also re-marks its control socket close-on-exec and scrubs its environment at startup, so neither the privileged transport nor a caller-controlled variable can leak into the kennel.

**Response.** On the same socketpair, the privhelper returns the **init host pid** and the **workload host pid** (kenneld needs the init host pid for the binder lifecycle identity gate — `sender_pid == init_host_pid`, design §7.2.2). The privhelper stays the init child's parent, reaps it, and **relays the exit status** back to kenneld over this socketpair when the kennel terminates — the reliable exit path is the process chain, not binder, which may already be torn down. Binder carries in-life lifecycle telemetry only.

The same `kennel-lib-spawn::wire` encoding is reused on two boundaries: as the construction-half body of this op, and as the supervision-half payload of the binder `BC_REPLY` to `GET_SANDBOX_PLAN`. Binder **copies** the reply buffer into the target's mmap'd region (cross-instance transactions reject `BINDER_TYPE_PTR`), so the Plan is forced into a localized flat buffer with no shared-memory hazard.

### `SpawnKennel` control op (roadmap)

Spawning a kennel on behalf of a running kennel is a **kenneld control-socket** operation (a privileged kenneld op, *not* a binder service transaction), added to the CLI↔kenneld control protocol as op 5 `SpawnKennel` / response tag 6 `Spawned`, reached from inside a kennel only when policy grants it (`[ipc.spawn]`). It is **roadmap** — designed in §7.1.7, contract in [`02-4-binder.md`](02-4-binder.md) §Kennel spawning — and will be merged into the request/response tables above when it lands. It carries no fds and applies the same `SO_PEERCRED` check and field bounds as every other control request.

Binder itself is a **separate IPC surface** from the control and privhelper protocols documented in this chapter: the per-instance binderfs bus (kenneld as node 0) that carries the service registry, the protocol facades (`org.projectkennel.IAfUnix`, …), the `kennel-bin-init` lifecycle control plane, and (roadmap) the `INet` network crossing. Its wire contract — the `BC_*`/`BR_*` ABI, the reserved-node verbs, and the cross-instance relay — lives in [`02-4-binder.md`](02-4-binder.md) and [`02-5-binder-net.md`](02-5-binder-net.md), not here.

### Locking

Status: not yet built (roadmap). As designed, the helper takes `flock` on `/run/kennel/privhelper.lock` (mode 0600, owned by root) before any privileged syscall, so concurrent invocations serialise with the lock held for the duration of one operation. As built, the helper takes no inter-process lock: each invocation runs its single validated operation directly, and concurrent invocations are not serialised at this layer.

## kenneld ↔ per-kennel daemons

### netproxy (no control socket — configured by file, reconfigured by respawn)

The egress proxy has **no control socket and no on-socket control protocol**. `host-netproxy` takes a single config-file path as `argv[1]`, loads it once at startup (`config::load`), binds one `TcpListener` per configured loopback address, and serves until a listener fails. kenneld writes the per-kennel egress TOML (`kenneld::proxy`) before spawning the proxy.

**Live-reload is built, via the config file — not a control socket.** A background thread (`spawn_reloader` in the binary) re-stats the config path every second; when its mtime changes it re-runs `config::load` and swaps the proxy's ruleset, host-services, and resolved-address opinion in place (`Proxy::reload`). The config-derived state lives behind an `RwLock<Arc<ProxyState>>` and each request snapshots the current `Arc` once, so the swap never restarts the proxy, never drops in-flight connections, and cannot split a single decision. A reload that fails to parse is logged and ignored — the running policy is kept, never silently widened. So kenneld reconfigures egress by **rewriting the config file** (no respawn needed for ruleset/host-service changes); only listen-address and audit-sink changes still need a respawn (those are not hot-reloaded). There is still no `config.load` / `config.reload` / `drain` / `status` method and no runtime status struct over a socket; per-request egress decisions are reported through the unified audit writer (`net.egress` records), not queried on a control socket. That on-socket control surface remains roadmap.

### dbus-proxy

Project Kennel uses `xdg-dbus-proxy` as an external dependency. Its control surface is not Project Kennel's; the kenneld interaction with it is via process signals (`SIGTERM` for drain) and the xdg-dbus-proxy filter file (regenerated on policy reload).

### SSH bastion (`kennel-sshd` / `kennel-akc`)

SSH egress is re-originated through a per-user `kennel-sshd` bastion (stock OpenSSH `sshd`), not an in-kennel agent (design §7.10). kenneld manages the bastion's lifecycle (spawn, signal, reap) and owns the forced-command bindings in memory. The one IPC surface is the control socket: the root-owned `AuthorizedKeysCommand` `kennel-akc` issues a `Request::AuthorizedKeys` (the offered key `%t %k`) to the running kenneld, which matches it against the live edge set (`Bastion::authorized_keys_for`) and returns the bound forced-command line(s); a non-match returns nothing and sshd authorises nothing. No `authorized_keys` file is written.

## Protocol invariants

For every protocol described here:

- The server validates every field of every request before acting. Unknown fields are rejected.
- String fields have explicit length caps documented in the schema; reads are bounded.
- The server never reflects untrusted strings back to the client without sanitisation (per CODING-STANDARDS.md §10.3).
- Numbers outside their declared range are rejected at parse time, not at semantics time.
- Errors are first-class responses, not connection drops. The server only drops the connection on:
  - Protocol-framing violation (length exceeds `MAX_MESSAGE`, or a body field is truncated, oversized, or non-UTF-8).
  - Client identity check failure (`SO_PEERCRED` UID mismatch).
  - The client closing the connection (or a closed/malformed frame). The server sets no read timeout; there is no idle-connection timeout.
