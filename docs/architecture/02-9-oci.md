# API surfaces — OCI substrate execution

This chapter is the implementation contract for OCI substrate execution, designed
in [`07-11-oci-substrate.md`](../design/07-11-oci-substrate.md) (§7.11). Where the
design chapter says *what and why* — boot an unpacked image as the kennel root,
split fetch from run so no parser reaches the daemon, treat the image as a loud
declared trust (T3.8) — this one commits to the concrete shape: the named store
layout, the OCI-model policy schema, the `kennel oci` verbs, the launcher binary
and its env denylist, and the bounded spawn-path branch that boots an image root.

> **Status: design complete; build sequenced (roadmap W17–W19).** The
> substrate-replacement plumbing this builds on already exists: `Plan.new_root`
> (`kennel-lib-spawn/src/plan.rs`) is a real `Option<PathBuf>`, wired through
> `wire.rs` and set by `kenneld` at bring-up (`lib.rs`), and
> [`build_view_and_pivot`](../../src/crates/kennel-lib-spawn/src/lib.rs) already
> takes `new_root` and `pivot_root`s onto it. What is **roadmap** is the OCI
> branch of that path (boot the image tree as `new_root` instead of a constructed
> tmpfs), the `[rootfs]` schema + grammar partition in `kennel-lib-compile`, the
> `kennel oci` verbs and the named store, and the `kennel-bin-oci-entry` launcher.
> Build order is §"Build sequencing" below. This chapter is reconciled to code as
> each item lands.

## The named-image store

`kennel oci build <name>` populates an operator-owned store entry:

```
<store>/<name>/
  rootfs/            # the unpacked image filesystem
  config.json        # the image's runtime config (OCI image-config blob)
  digest             # the resolved image@sha256 the build pulled from
  policy.toml        # the scaffolded run policy (operator completes + signs)
```

The store root defaults under the operator's data dir
(`~/.local/share/kennel/images/`) via the config cascade — no hardcoded path — and
is per-operator (the per-user `0700` home is the isolation boundary; no shared
store). `kennel oci run <name>` resolves `<name>` against the store, not a path.

**Integrity unit.** `rootfs/`, `config.json`, and `digest` are read-only at run
and form one integrity unit (the launcher trusts `config.json` for the entrypoint
and env, so a post-build edit of it redirects execution as effectively as
tampering with the rootfs). `policy.toml` is outside the unit — it is
signature-covered separately, and the signature is what authorises the run. The
integrity ladder (design §7.11.8) hashes/verities the three-file unit, not the
rootfs alone.

## The OCI-model policy schema

```toml
[rootfs]
# The named store entry's rootfs, resolved by `kennel oci run <name>`. Absolute or
# home-relative. Its presence marks the policy OCI-model.
path   = "~/.local/share/kennel/images/my-app/rootfs"
# The image@sha256 the build pulled from, recorded by `kennel oci build`.
# Signature-covered; the runner refuses unless it equals the store `digest` file.
image  = "ghcr.io/my-org/my-app@sha256:<digest>"
# Required: the substrate-trust waiver (T3.8) is loud.
reason = "run vendor image my-app; content unvetted by construction"
```

**Grammar partition (compiler rule).** `kennel-lib-compile` enforces that
`[rootfs]` is present **iff** the policy is consumed by `kennel oci run`:
`kennel run` rejects a policy containing `[rootfs]`, and `kennel oci run` rejects
one without it. `[rootfs]` without `reason` is refused. The substrate-trust
exposure is *derived* from the grant (no stored threat-tag field) and surfaced by
`kennel policy risks` with the grant as carrier and the operator's `reason` — the
same derivation path `mode = host` uses for T1.6.

There is no `append_args` field and no OCI-specific `[workload]` shape. The
entrypoint comes from the image config via the launcher, or from an explicit
`[workload].argv` override (design §7.11.5) which opts out of the entire image
config and is pinned in-root like a standard `kennel run`.

## The `kennel oci` verbs

`kennel oci build <name> -- <image-ref>` runs the **Kennel-shipped vetted fetch
policy** (a `constrained`-egress, registry-allowlisted, `fs.write`-to-the-store
policy the operator does not author) in a confined kennel running `skopeo`/`umoci`;
unpacks `rootfs/` + `config.json`; resolves a tag reference to a digest at build
time and records `digest`; and scaffolds `policy.toml` (the confined default + a
`[rootfs]` block with the recorded `image` + `reason = "…"`). The registry grant
surfaces loudly (`build foo -- ghcr.io/…` reports the `ghcr.io:443` egress against
the configured registry set).

`kennel oci run <name>` resolves the store entry, verifies the signed `policy.toml`
against the trust store, asserts `policy.image == store/digest` (refuses on
mismatch), and drives the standard `kennel run` bring-up with the image-root
spawn-path branch below.

## `kennel-bin-oci-entry` — the launcher

A new binary in the `kennel-bin-*` class (neither end of a binder connector, like
`kennel-bin-init`): `#![forbid(unsafe_code)]`, minimal deps, **workload-side** —
it runs at the workload's authority and is **not** in the daemon TCB (it must not
appear in `cargo tree -p kenneld`). `kenneld` binds it to a known path in the view
and the image's `config.json` read-only (e.g. `/run/kennel/oci-config.json`), and
makes it `argv[0]`. Post-`pivot_root` it:

1. reads `config.json`;
2. `chdir`s to `WorkingDir`;
3. **sanitises** then merges `Env` (below);
4. resolves `Entrypoint` + `Cmd` and `execve`s the image entrypoint, resolved
   in-root (it holds nothing to drop).

### `env_strip` — the image-`Env` denylist

The launcher's `env_strip` module is the *only* gate on image-supplied
environment (no policy field reaches that input — it is orthogonal to `[env].deny`,
which filters the caller→kennel pass-through). The strip runs **before** the merge;
Kennel's synthesised env is layered on top unfiltered, so `[env].set` re-adds any
stripped name and wins. The denylist (the `AT_SECURE`-equivalent set):

- **Prefix-strip** `LD_*`, `GLIBC_*` (future-proof against new loader/tunable vars).
- **Tier 1 — glibc loader floor** (non-`LD_*` `unsecvars.h`): `GCONV_PATH`,
  `GETCONF_DIR`, `HOSTALIASES`, `LOCALDOMAIN`, `LOCPATH`, `MALLOC_TRACE`,
  `NIS_PATH`, `NLSPATH`, `RESOLV_HOST_CONF`, `RES_OPTIONS`, `TMPDIR`, `TZDIR`.
- **Tier 2 — language runtimes**: `NODE_OPTIONS`, `NODE_PATH`, `PYTHONPATH`,
  `PYTHONHOME`, `PYTHONSTARTUP`, `PERL5LIB`, `PERL5OPT`, `PERLLIB`, `RUBYLIB`,
  `RUBYOPT`, `CLASSPATH`, `JAVA_TOOL_OPTIONS`, `_JAVA_OPTIONS`, `JDK_JAVA_OPTIONS`.
- **Tier 3 — shell-entrypoint sourced files**: `BASH_ENV`, `ENV`, `SHELLOPTS`,
  `BASHOPTS`.

`PATH` and `IFS` are deliberately **not** stripped (`PATH` is the image floor that
policy overrides; `IFS` is reset by modern shells). The wire-in test: the sanitise
is added to the merge in the *same* change — a bare merge silently reopens the hole
and nothing fails unless a test asserts it; the `env_strip` behavioural tests
(image loader/runtime/shell vars gone, policy re-add survives) ship with it.

## Daemon spawn-path branch (the bounded `kenneld` change)

`[rootfs]` flows `loaded.rootfs` → `Spec` → `kennel-lib-spawn::Plan.new_root`. The
non-OCI path sets `new_root` to a fresh constructed-view tmpfs; the OCI path sets
it to the image `rootfs/`. [`build_view_and_pivot`](../../src/crates/kennel-lib-spawn/src/lib.rs)
branches on this:

- **No tmpfs scaffold.** The non-OCI path `mount_special("tmpfs", new_root)`; for an
  image root the tree already exists, so it is bind-mounted to itself to become a
  mount point instead.
- **No merged-usr mirroring.** The `for link in ["bin","sbin","lib",…]` host-symlink
  mirror is skipped — the image carries its own layout (may be musl/busybox).
- **Targeted `/etc` overlay, not synthetic `/etc`.** Instead of a fresh `/etc` dir
  populated from `file_binds`, `/etc` is a small kennel-owned `tmpfs` **seeded by
  copying the image's `/etc`**, with Kennel's handful written on top:
  `resolv.conf`, `nsswitch.conf`, `hosts` (network); `hostname`; `passwd`, `group`
  (persona). Each is a whole-file replacement. Two fixed rules: copy image `/etc`
  symlinks **verbatim** (`symlinkat`, never dereference — the copy runs as
  `kenneld`, so a crafted `/etc/x → ../../../<host>` must not pull host content in);
  and write Kennel's files **unlink-then-create** (an image `resolv.conf` symlink
  would otherwise be followed out of the tmpfs), then read-only-bind them over the
  writable tmpfs. **Nothing is written to `rootfs/`** — it is read-only for real, so
  the integrity-ladder hash/verity is never invalidated by the runner's own writes.
- **Masks, `/dev`, `/proc`, `/tmp`** layer onto the image tree as today; the masks
  still cover additive writable binds. The constructed-home chown discriminator
  (device id) already excludes the operator-owned image tree.
- **In-root entrypoint resolution.** The image entrypoint is resolved with
  `openat2(RESOLVE_IN_ROOT)` beneath the rootfs and `execve`d on the fd (fd-pin
  holds across pivot). Escape to host `/` is closed by the post-pivot old-root
  detach; the residual `RESOLVE_IN_ROOT` does *not* close is an image symlink (or a
  surviving loader path) resolving into an additive `[fs.read/write]` bind — the
  same confused-deputy seam any writable bind has, mitigated by the bind-source
  guard, not OCI-specific.
- **Digest equality check.** Before boot the runner asserts the signed
  `[rootfs].image` equals the store entry's `digest` file and refuses on mismatch.

## Build sequencing

Maps design §7.11 / the roadmap (Thrust 5). Unprivileged surface first, then the
gated daemon change, then opt-in hardening.

1. `kennel oci build`/`run` verbs + the named store manager (CLI, unprivileged).
2. `[rootfs]` schema + `kennel-lib-compile` grammar partition + the T3.8 risk
   derivation.
3. The vetted fetch policy + the `TEMPLATE-oci` run-policy scaffold. The scaffold
   mirrors the `env_strip` denylist as `[env].deny` globs so the strip shows in the
   signed policy and `kennel policy risks` (not enforcement — the launcher is the
   gate; the module is authoritative if they drift).
4. `kennel-bin-oci-entry` (config parse, env sanitise + merge, `chdir`, in-root
   `execve`), with the wire-in invariant above.
5. The T3.8 THREATS entry (done) and the compliance-table mapping under supply
   chain (NIS2 21(2)(d), DORA Art. 28).
6. **The daemon branch** (gated, sign-off given): `[rootfs]` → `Spec` →
   `Plan.new_root`; the `build_view_and_pivot` branch; the `/etc` overlay; in-root
   resolution; the digest equality check.
7. Integrity Rung 1 (content-addressed entry) and Rung 2 (fs-verity), opt-in,
   behind the floor.
