# API surfaces — OCI substrate execution

This chapter is the implementation contract for OCI substrate execution, designed
in [`07-11-oci-substrate.md`](../design/07-11-oci-substrate.md) (§7.11). Where the
design chapter says *what and why* — boot an unpacked image as the kennel root,
split fetch from run so no parser reaches the daemon, treat the image as a loud
declared trust (T3.8) — this one commits to the concrete shape: the named store
layout, the OCI-model policy schema, the `kennel oci` verbs, the launcher binary
and its env denylist, and the bounded spawn-path branch that boots an image root.

> **Status: built — the layered-overlay vertical (design §7.11.4a) is in tree and
> green.** Verbs (`build`/`run`/`revert`/`update`) + named store + grammar partition
> (`[rootfs]` XOR `[workload]`) + digest check; the `kennel-bin-oci-entry` launcher
> (env strip + `-- <cmd>` Cmd override); the W18 producer (`RootfsRuntime` with
> `persistence`); the daemon launcher binding (`OciPrep`); and the three-lower overlay
> spawn path (`kennel-etc : image : scaffold`, the binary `persistence` upper, `/etc`
> writable-through, the `st_dev` chown fix, the scaffold built in the staging tmpfs); and
> **closure-lock** (§7.11.4c) — the build-derived `[rootfs].readonly`/`writable` read-only
> mounts (+ a `/` rwx Landlock substrate) that re-impose the executable boundary the
> DAC-flatten erased; the **confined fetch** (`skopeo`/`umoci` under the signed
> `oci-fetch@v1` view); **`update`** re-deriving the base closure-lock from the new image
> while preserving the operator's carve-outs and clearing the signature; and **`revert`**'s
> diff-against-pin (`--list`) + selective restore (`-- <path>`).
> **Roadmap:** the OCI integrity ladder (fs-verity / content-addressed rung) — the
> digest-pinned floor is the minimum; the ladder is a later release.

## The named-image store

`kennel oci build <name>` populates an operator-owned store entry:

```
<store>/<name>/
  rootfs/            # the unpacked image filesystem (the overlay's read-only lower)
  config.json        # the image's runtime config (OCI image-config blob)
  digest             # the resolved image@sha256 the build pulled from
  policy.toml        # the scaffolded run policy (operator completes + signs)
  upper/  work/      # the managed overlay upper (persistence = "persist" only)
```

The store root defaults under the operator's data dir
(`~/.local/share/kennel/images/`) via the config cascade — no hardcoded path — and
is per-operator (the per-user `0700` home is the isolation boundary; no shared
store). `kennel oci run <name>` resolves `<name>` against the store, not a path.
`upper/` + `work/` exist only for a `persistence = "persist"` entry, are outside the
integrity unit, and `work/` is outside the T2.8 masks.

**Integrity unit.** `rootfs/`, `config.json`, and `digest` are read-only at run
and form one integrity unit (the launcher trusts `config.json` for the entrypoint
and env, so a post-build edit of it redirects execution as effectively as
tampering with the rootfs). `policy.toml` is outside the unit — it is
signature-covered separately, and the signature is what authorises the run. The
integrity ladder (design §7.11.8) hashes/verities the three-file unit, not the
rootfs alone.

## The OCI-model policy schema

```toml
[rootfs]
# The named store entry's rootfs, resolved by `kennel oci run <name>`. Absolute or
# home-relative. Its presence marks the policy OCI-model.
path   = "~/.local/share/kennel/images/my-app/rootfs"
# The image@sha256 the build pulled from, recorded by `kennel oci build`.
# Signature-covered; the runner refuses unless it equals the store `digest` file.
image  = "ghcr.io/my-org/my-app@sha256:<digest>"
# Required: the substrate-trust waiver (T3.8) is loud.
reason = "run vendor image my-app; content unvetted by construction"
# Rootfs persistence (§7.11.4a): "discard" (default) | "persist" (binary). "persist" is
# a loud value — kennel policy risks derives its exposure.
persistence = "discard"
# Closure-lock (§7.11.4c): build-derived for a non-root image, signed here. Read-only
# MOUNTS over rootfs paths (read+execute kept; write ⇒ EROFS); longest-prefix wins via
# mount nesting. ["/"] = immutable.
readonly = ["/usr", "/lib"]
# writable = ["/usr/lib/python3.12"]   # carve a hole back out; loud, derived risk
```

**Closure-lock (split: Landlock substrate + read-only mounts).** The unprivileged `oci
build` flattens every image inode to the persona uid, so the rootfs is writable by default
(the flatten made DAC vacuous — honest labelling, not a policy assertion). The flatten follows
from the **userns single-entry map**: a Kennel userns maps exactly one entry — the host operator
identity to one in-namespace persona uid, with no subuid range — so no foreign image uid is
representable, and `config.User` is read only at build (for the closure-lock decision below),
never honoured as a runtime uid; an image that *assumes* its uid (a `gosu 1000` drop, an `id -u`
check) breaks on identity, not permission, because the flatten already gave the persona ownership
of the whole tree. The Plan
constructor's OCI base is a single Landlock grant — `/` read+write+execute — the broad
writable substrate. The closure itself is **not** Landlock: Landlock access rights are
*additive*, so a `/` write cannot be subtracted at `/usr` (proven on hardware — the write
succeeded). Instead `build_image_view_and_pivot` makes each `[rootfs].readonly` path a
**read-only mount** over the merged root (`bind` + `remount_readonly`), so a write fails
`EROFS`; each `[rootfs].writable` carve-out is bound read-write first, so its deeper
mountpoint serves its own subtree (longest-prefix wins by mount nesting). The constructed
mounts (`/tmp`, the persona `/home`, `/dev`) and the read-only `[fs.read]` binds / T2.8
masks are their own mounts, so neither the broad Landlock grant makes the read-only ones
writable, nor does `readonly = ["/"]` strand the writable ones. The lock is robust because
the persona workload holds no `CAP_SYS_ADMIN` (not in-ns root) and `mount` is
seccomp-blocked. The `readonly` set is **derived at `oci build`** from `config.User`
(`derive_closure_readonly`: non-root ⇒ the FHS closure `/usr` `/bin` `/sbin` `/lib*`;
all-root ⇒ none) and written into the scaffolded policy — operator-reviewed,
signature-ratified, no compiler default. `ImageRoot` carries the two lists to the seal.

**Grammar partition (compiler rule).** `kennel-lib-compile` enforces that `[rootfs]`
and `[workload]` are **mutually exclusive at the block level**, and that `[rootfs]`
is present **iff** the policy is consumed by `kennel oci run`: `kennel run` rejects a
policy containing `[rootfs]`; `kennel oci run` rejects one without it; an OCI policy
carrying a `[workload]` block is refused at compile. `[rootfs]` without `reason` is
refused. The substrate-trust exposure is *derived* from the grant (no stored
threat-tag field) and surfaced by `kennel policy risks` with the grant as carrier and
the operator's `reason` — the same derivation path `mode = host` uses for T1.6;
`persistence = "persist"` derives a second, distinct exposure the same way.

There is no `[workload]` block and no per-binary pin in the OCI model: the entrypoint
always comes from the image config via the launcher (provenance is anchored on the
image digest). A `kennel oci run <name> -- <cmd>` is a runtime `Cmd` override the
launcher applies in place of the image's `Cmd` (keeping the image `Entrypoint` + the
sanitised `Env`) — no policy impact, so it crosses no signature.

## The `kennel oci` verbs

`kennel oci build <name> -- <image-ref>` runs the **Kennel-shipped vetted fetch
policy** (a `constrained`-egress, registry-allowlisted, `fs.write`-to-the-store
policy the operator does not author) in a confined kennel running `skopeo`/`umoci`;
unpacks `rootfs/` + `config.json`; resolves a tag reference to a digest at build
time and records `digest`; and scaffolds `policy.toml` (the confined default + a
`[rootfs]` block with the recorded `image` + `reason = "…"`). The registry grant
surfaces loudly (`build foo -- ghcr.io/…` reports the `ghcr.io:443` egress against
the configured registry set).

`kennel oci run <name>` resolves the store entry, verifies the signed `policy.toml`
against the trust store, asserts `policy.image == store/digest` (refuses on
mismatch), and drives the standard `kennel run` bring-up with the image-root
spawn-path branch below.

`kennel oci revert <name>` restores the managed upper toward the image lower. The image
lower is the **pin** (content-addressed by its `digest`); the upper's copy-ups and
whiteouts are the **diff against the pin**; removing an upper entry is **restore-from-pin**
(the lower shows back through) — the OCI instantiation of the same pin / diff-against-pin /
restore-from-pin mechanism as the masked trust-manifest store (§7.4, `.trust-manifest.d`),
which pins workspace bytes the same way. `--list` prints the diff (`M` a copy-up/added file,
`D` a whiteout deleting a lower file); `-- <path>…` selectively restores the named paths;
with neither it is the **total** case — obliterate the whole `upper/` + `work/` (the blunt
end of selective revert), a no-op for a `discard` entry (there is no managed upper; `readonly`
is a closure-lock fact on a different axis, not a persistence value). `--list` is read-only
and allowed while running; the mutating modes refuse while running. The image lower is never
touched (revert returns the *mutable* state toward the pin, it does not re-attest the image).
`kennel oci update
<name> -- <ref>` re-runs the builder for the new image, swaps `rootfs/` +
`config.json` + `digest`, bumps `[rootfs].image`, **re-derives the base closure-lock
from the new image while preserving the operator's hand-added carve-outs** (§"Closure
re-derivation on update" below), **clears the policy signature** (the operator
reviews + re-signs), and discards the upper by default (`--keep-state` preserves it
with a rebase-hazard warning). Both refuse while the entry is running; `update`
refuses an absent `<name>`, as `build` refuses a present one.

**Closure re-derivation on update.** The operator may have hand-tuned the previous
policy's lists — a `writable = ["/usr/lib/python3.12"]` carve-out for `pip`, a
`readonly = ["/opt/app"]` over a vendored tree. `update` must not silently drop them:
it replaces only the **build-derived base** closure (the FHS set, or none for an
all-root image) and **preserves the operator's carve-outs** — the `writable` list
verbatim, and any `readonly` entry not in the base — then surfaces the before/after
diff at re-sign. Because `update` also clears the signature, a silently-regressed lock
would otherwise be re-signed without the operator noticing; the diff is what makes the
re-sign a real review.

## `kennel-bin-oci-entry` — the launcher

A new binary in the `kennel-bin-*` class (neither end of a binder connector, like
`kennel-bin-init`): `#![forbid(unsafe_code)]`, minimal deps, **workload-side** —
it runs at the workload's authority and is **not** in the daemon TCB (it must not
appear in `cargo tree -p kenneld`).

> **Static linking is mandatory for every binary the seal runs *inside* the kennel
> root** — `kennel-bin-init`, `kennel-bin-oci-entry`, and the in-kennel facades
> (`facade-socks5`/`afunix`/`ssh`/`dbus`/`client`). For an OCI substrate that root is
> an arbitrary image's filesystem (musl, distroless, or `scratch`), which has no host
> `ld.so`/`libc`, so a dynamically-linked host binary fails to load there
> (`libgcc_s.so.1: cannot open shared object`). `install.sh` builds these with glibc
> `+crt-static` (a `static-pie` with no `PT_INTERP` and no `NEEDED` libs — no source
> change) into `target/<triple>/release` and installs them from there; the host-side
> delegates (`kenneld`, the CLI, `host-netproxy`/`inetd`/`dbus`, the privhelper) run in
> the operator's context where host libc is present and stay dynamic. (musl would be
> cleaner but `kennel-lib-binder`'s `ioctl` request type differs glibc↔musl, so the
> binder-using in-kennel set stays glibc-static until that is made portable.) `kenneld` binds it to a known path in the view
and the image's `config.json` read-only (e.g. `/run/kennel/oci-config.json`), and
makes it `argv[0]` (with the config path as `argv[1]`, and any `kennel oci run -- <cmd>`
override tokens after it). Post-`pivot_root` it:

1. reads `config.json`;
2. `chdir`s to `WorkingDir`;
3. **sanitises** then merges `Env` (below);
4. resolves `Entrypoint` + `Cmd` — using the override tokens in place of the image's
   `Cmd` when present, the image `Entrypoint` always kept — and `execve`s the
   entrypoint, resolved in-root (it holds nothing to drop).

### `env_strip` — the image-`Env` denylist

The launcher's `env_strip` module is the *only* gate on image-supplied
environment (no policy field reaches that input — it is orthogonal to `[env].deny`,
which filters the caller→kennel pass-through). The strip runs **before** the merge;
Kennel's synthesised env is layered on top unfiltered, so `[env].set` re-adds any
stripped name and wins. The denylist (the `AT_SECURE`-equivalent set):

- **Prefix-strip** `LD_*`, `GLIBC_*` (future-proof against new loader/tunable vars).
- **Tier 1 — glibc loader floor** (non-`LD_*` `unsecvars.h`): `GCONV_PATH`,
  `GETCONF_DIR`, `HOSTALIASES`, `LOCALDOMAIN`, `LOCPATH`, `MALLOC_TRACE`,
  `NIS_PATH`, `NLSPATH`, `RESOLV_HOST_CONF`, `RES_OPTIONS`, `TMPDIR`, `TZDIR`.
- **Tier 2 — language runtimes**: `NODE_OPTIONS`, `NODE_PATH`, `PYTHONPATH`,
  `PYTHONHOME`, `PYTHONSTARTUP`, `PERL5LIB`, `PERL5OPT`, `PERLLIB`, `RUBYLIB`,
  `RUBYOPT`, `CLASSPATH`, `JAVA_TOOL_OPTIONS`, `_JAVA_OPTIONS`, `JDK_JAVA_OPTIONS`.
- **Tier 3 — shell-entrypoint sourced files**: `BASH_ENV`, `ENV`, `SHELLOPTS`,
  `BASHOPTS`.

`PATH` and `IFS` are deliberately **not** stripped (`PATH` is the image floor that
policy overrides; `IFS` is reset by modern shells). The wire-in test: the sanitise
is added to the merge in the *same* change — a bare merge silently reopens the hole
and nothing fails unless a test asserts it; the `env_strip` behavioural tests
(image loader/runtime/shell vars gone, policy re-add survives) ship with it.

## Daemon spawn-path branch — the layered overlay

`[rootfs]` flows `loaded.rootfs` → `Spec` →
`kennel-lib-spawn::ShimView` (the `image: Option<ImageRoot>` field, carrying the
lower paths + the persistence mode). When `Some`,
[`build_view_and_pivot`](../../src/crates/kennel-lib-spawn/src/lib.rs) takes the
overlay head instead of the constructed-tmpfs head; both share `seal_view_tail`.

**The lower stack** — read-only, leftmost wins:

```
lowerdir = kennel-etc : image-rootfs : scaffold
```

- **`kennel-etc`** — a small Kennel-owned dir kenneld builds per run from the synthetic
  `/etc` files (the `file_binds`): `resolv.conf`, `hostname`, `passwd`, `group`, `hosts`,
  `nsswitch.conf`. Highest precedence, so Kennel's resolver + persona win over the image
  — and because it is layer precedence, not a copy, a real Kennel file outranks an image
  *symlink* at the same path with **no unlink-replace and no dereference** (this retires
  the seed-copy hazard the single-lower version carried).
- **`image-rootfs`** — the unpacked image, read-only, digest-pinned, ladder-covered.
  **Never an upperdir** — that would make the image writable and land workload writes in
  the hashed entry, destroying the integrity unit.
- **`scaffold`** — a Kennel-shipped read-only dir (`Deployment::oci_scaffold`, default
  `<libexec>/oci-scaffold`) of empty mountpoint dirs (`/proc`, `/dev`, `/tmp`, `/sys`,
  `/run/kennel`, `/etc`) and empty placeholders for the `kennel-etc` targets, so a
  distroless/scratch image that ships none of them still boots (esp. under `readonly`).
  Lowest precedence; the image shows through wherever it ships a real path.

**The upper is the persistence binary** (`ShimView.image.persistence`) — there is **always**
an upper (whole-tree immutability is `readonly = ["/"]` via the closure-lock read-only mounts, not a
no-upper mount):

- `discard` — an ephemeral `tmpfs` upper (under the kenneld staging dir), gone at teardown.
- `persist` — a managed upper at `<store>/<name>/upper` (+ `work/`); kenneld creates them.
  The store fs must carry `user.*` xattrs (the `userxattr` whiteouts an unprivileged-userns
  overlay needs); a fs that drops them fails the overlay mount clearly. `discard`'s tmpfs
  upper relies on the same support universally (tmpfs user xattrs, kernel ≥ 6.6 — under the
  6.10 floor); the explicit pre-check is `persist`-only (the store is the operator-influenced
  target).

**Assembly** (`build_image_view_and_pivot`), over the merged root in order: a tmpfs `/home`
(so the persona home is writable independent of the rootfs closure-lock); the granted `~/` +
launcher + `config.json` + additive `[fs]` binds (which win at their mountpoints). **`/etc` is
writable-through — nothing is ro-bound** (§7.11.4c): the `kennel-etc` lower sets the defaults,
a copy-up harms only the workload's own view, and enforcement is the netns/BPF/uid-map/
Landlock/seccomp, never file content. Then the seal tail: construct `/dev`, fresh `/proc`,
private `/tmp`, T2.8 masks (sourced from the view's `/tmp`, writable in every mode),
`pivot_root` (`put_old` is the scaffold's `/.kennel-oldroot`). No merged-usr mirror.

**The `st_dev` chown fix.** The constructed-home chown discriminator keyed on `/`'s device,
but the OCI persona home is on the dedicated `/home` tmpfs (a different device from the merged
overlay `/`), so it would be wrongly skipped. The walk now keys on the **home's own device**
and guards the whole-home-bind case by **ownership** (a bind is already operator-owned), which
is correct for both the constructed view and the OCI overlay.

**`mount_overlay`** (`kennel-lib-syscall/src/mount.rs`) takes an ordered lower list and an
optional `(upper, work)`; overlayfs is unprivileged-userns-mountable (kernel ≥ 5.11), so it
mounts in the kennel's own userns with no real privilege. In-root entrypoint resolution is
inherent — the launcher `execve`s after `pivot_root`, so the kernel resolves within the
pivoted root, no daemon-side `openat2`. The digest equality check
(`[rootfs].image == store/digest`) is the **runner's** (`kennel oci run`), not the daemon's.

## Build sequencing

Maps design §7.11 / the roadmap (Thrust 5). Unprivileged surface first, then the
gated daemon change, then opt-in hardening.

1. `kennel oci build`/`run` verbs + the named store manager (CLI, unprivileged).
2. `[rootfs]` schema + `kennel-lib-compile` grammar partition + the T3.8 risk
   derivation.
3. The vetted fetch policy + the `TEMPLATE-oci` run-policy scaffold. The scaffold
   mirrors the `env_strip` denylist as `[env].deny` globs so the strip shows in the
   signed policy and `kennel policy risks` (not enforcement — the launcher is the
   gate; the module is authoritative if they drift).
4. `kennel-bin-oci-entry` (config parse, env sanitise + merge, `chdir`, in-root
   `execve`), with the wire-in invariant above.
5. The T3.8 THREATS entry and its compliance-table mapping under supply chain
   (NIS2 21(2)(d), DORA Art. 28) — done.
6. **The daemon branch** (gated, sign-off given; **built, then reworked to the
   layered overlay**): `[rootfs]` → `RootfsRuntime` → `ShimView.image`; the
   `build_image_view_and_pivot` three-lower stack (`kennel-etc : image : scaffold`)
   + the binary `persistence` upper + `/etc` writable-through + the `st_dev` chown
   fix, sharing `seal_view_tail`; **closure-lock** in the Plan constructor + its
   build-time derivation. The digest equality check rides the runner; in-root
   resolution is inherent to the post-pivot launcher `execve`. Adds `oci revert`/`update`.
7. Integrity Rung 1 (content-addressed entry) and Rung 2 (fs-verity), opt-in,
   behind the floor.
