# API surfaces ‚Äî CLI

## Stability commitment

**Stable** per `02-0-overview.md`. Subcommands, flags, and exit codes are backwards-compatible across minor versions. Human-readable text output is best-effort and may evolve for clarity; scripts must not parse the human format.

Deprecations follow the policy in `02-0-overview.md` ¬ßDeprecation policy: announced in CHANGELOG, warned at runtime, minimum one minor version of overlap before removal.

---

## Invocation

```
kennel <SUBCOMMAND> [ARGUMENTS]
```

The CLI is the user's entry point. It is stateless: every invocation starts from disk and either contacts kenneld over its control socket (`run`, `stop`, `list`) or operates purely on local files (`compile`, `validate`, `sign`, `keygen`, `audit`). `kennel` does not maintain a long-running connection across invocations.

The CLI dispatches on the first argument; each subcommand parses its own flags. The kenneld control socket is `/run/user/<uid>/kennel/control.sock` (the `kenneld.socket` user unit binds it; kenneld is socket-activated on the first `run`).

## Subcommands

The implemented verbs are `run`, `stop`, `list`, `compile`, `validate`, `sign`, `keygen`, and `audit`.

### `kennel run` ‚Äî start a workload inside a kennel

```
kennel run <policy> <name> -- <command> [args...]
```

Asks kenneld to start a kennel named `<name>` from the settled policy at `<policy>` and run `<command>` inside it. The `--` separator delimits the kennel arguments from the workload command. The command after `--` must be non-empty.

The kennel runs in the foreground. The CLI passes this terminal's three stdio file descriptors to kenneld over `SCM_RIGHTS`; **kenneld** (not the CLI) performs the spawn sequence, attaching the workload to those descriptors. The CLI blocks until the workload exits and then exits with the workload's exit code (clamped to a byte).

Protocol: kenneld first replies `Started { ctx, pid }` (the CLI prints a one-line confirmation to stderr), then `Exited { code }` when the workload terminates. An `Error` at either point is printed to stderr and the CLI exits non-zero.

Each `kennel run` is one kennel: it gets its own context byte, cgroup, egress proxy, loopback addresses, and constructed `$HOME` view, all torn down immediately when the workload exits.

### `kennel stop` ‚Äî stop a running kennel

```
kennel stop <name>
```

Asks kenneld to stop the named kennel. kenneld terminates the workload and tears down the kennel's per-kennel resources (proxy, loopback addresses, cgroup, constructed view) immediately. Replies `Stopped` on success; the CLI prints a confirmation and exits 0.

### `kennel list` ‚Äî list running kennels

```
kennel list
```

Asks kenneld for the running kennels and prints a table (`NAME`, `CTX`, `PID`, `STATE`). State is `running` once the workload is launched, otherwise `starting`. Prints `no running kennels` when the daemon reports none.

### `kennel compile` ‚Äî resolve a source policy into a settled policy

```
kennel compile <policy> [--output-path <path>] [--key <key> | --unsigned]
               [--require-signed] [--no-lock] [--template-dir <dir>]... [--trust-dir <dir>]...
```

Resolves a source policy fully ‚Äî template-chain fold, signed includes, leaf deltas, source-signature verification, lockfile byte-checks, invariant validation, and installation-constant substitution ‚Äî and writes a **settled policy**: a flat, canonical-TOML artefact the runtime consumes directly. See `02-2-config-schema.md` ¬ßThe settled policy and design doc ¬ß9.10. Stateless: `compile` never contacts the daemon.

One of `--key` or `--unsigned` is required (they are mutually exclusive): `--key` signs the settled policy with an ed25519 key; `--unsigned` seals it for development use only.

Flags:

- `--output-path <path>` ‚Äî where to write the settled policy. Default: `<policy-dir>/<name>.settled.toml`.
- `--key <path>` ‚Äî ed25519 signing key for the settled policy.
- `--unsigned` ‚Äî emit a content-sealed but unsigned settled policy (development use).
- `--require-signed` ‚Äî refuse unsigned templates during resolution; verify every template against the trust store.
- `--no-lock` ‚Äî skip writing/verifying the `<name>.lock` file beside the output.
- `--template-dir <dir>` ‚Äî additional template search path (repeatable). The user (`~/.config/kennel/templates`) and system (`/etc/kennel/templates`) directories are always searched.
- `--trust-dir <dir>` ‚Äî additional trust-store path holding `<key_id>.pub` keys (repeatable). The user (`~/.config/kennel/keys`) and system (`/etc/kennel/keys`) directories are always searched.

A `<name>.lock` file beside the output byte-pins the resolved references. A re-tagged or re-signed reference relative to a prior lockfile is an integrity failure (exit 6).

Exit codes: 3 for policy-validation or resolution failure, 6 for a source-signature or lockfile-pin failure.

### `kennel validate` ‚Äî validate a policy without emitting an artefact

```
kennel validate <policy> [--require-signed] [--template-dir <dir>]... [--trust-dir <dir>]...
```

Resolves and checks a policy (chain, signatures, deltas, includes, invariants) without writing a settled policy. Exits 0 if it resolves cleanly (printing a one-line summary with the reference and deferred-substitution counts); otherwise the same exit code `compile` would return for that failure.

Flags `--require-signed`, `--template-dir`, and `--trust-dir` behave as in `compile`.

### `kennel sign` ‚Äî sign a source template or fragment

```
kennel sign <template> --key <key> [--output <path>]
```

Signs a source template or fragment with an ed25519 key, **appending** a `[signature]` block to the file (the signature covers the canonical re-serialisation, not the raw bytes, so comments and formatting are preserved). Refuses a file that already carries a `[signature]`. Leaf policies need not be signed.

Prints the public key to install in the trust store as `<key_id>.pub`.

Flags:

- `--key <path>` ‚Äî required. The ed25519 signing key.
- `--output <path>` ‚Äî write the signed file here instead of overwriting the input.

### `kennel keygen` ‚Äî generate an ed25519 signing key

```
kennel keygen <key-id> [--dir <dir>] [--force]
```

Generates an ed25519 keypair from the OS CSPRNG and writes it into the user key
dir (default `$XDG_CONFIG_HOME/kennel/keys`, else `~/.config/kennel/keys`):
`<key-id>.key` is the base64 seed (mode `0600`) passed to `--key`
(`sign`/`compile`/`run`), and `<key-id>.pub` is the base64 public key. The
`<key-id>` is both the filename and the signature `key_id` (restricted to 1‚Äì64
chars of `[A-Za-z0-9._-]`). Refuses to overwrite an existing seed without
`--force` ‚Äî replacing a key invalidates everything signed with the old one.
Prints the exact command to install the public key into the system trust store
(`/etc/kennel/keys/<key-id>.pub`) so the daemon trusts policies signed with it.
Stateless: `keygen` never contacts the daemon.

Flags:

- `--dir <dir>` ‚Äî write the keypair here instead of the default user key dir.
- `--force` ‚Äî replace an existing `<key-id>.key` (and its `.pub`).

### `kennel audit` ‚Äî query a kennel's audit log

```
kennel audit <name> [--resource CLASS] [--since DUR] [--novel-only] [--follow] [--print-journalctl-command]
```

Reads the per-kennel JSONL audit files directly from the file sink
(`$XDG_STATE_HOME/kennel/<name>/<class>.jsonl`, falling back to
`~/.local/state/kennel/‚Ä¶`, ¬ß02-3) ‚Äî no daemon round-trip, queryable from a fresh
shell. Events from the selected class files are printed in `ts` order.

Flags:

- `--resource <class>` ‚Äî limit to one class: `net` (‚Üí `network.jsonl`), `fs`
  (‚Üí `filesystem.jsonl`), `exec`, `unix`, `dbus`, `priv`, or `lifecycle`. Default:
  all classes present.
- `--since <dur>` ‚Äî keep only events newer than the given age (`90s`, `30m`, `2h`,
  `7d`; a bare number is seconds). Compared against each line's RFC3339-UTC `ts`.
- `--novel-only` ‚Äî collapse repeats: lines identical but for their timestamp are
  shown once (their first occurrence), so a flood of identical denials reads as one.
- `--follow` ‚Äî after the initial dump, stream new events as they are appended
  (re-reading from the start if a file was rotated).
- `--print-journalctl-command` ‚Äî instead of reading files, print the equivalent
  `journalctl --user KENNEL_KENNEL=<name> [KENNEL_RESOURCE=<class>] [--since "<dur> ago"]`
  invocation, for journald-only deployments.

A `<name>` containing `/` or `..` is refused (it is a path component). A kennel with
no audit files yet prints a note and exits 0.

## Exit codes

| Code | Meaning |
|---|---|
| 0 | Success. For `kennel run`, the workload's own exit code is passed through (clamped to a byte). |
| 1 | Generic failure: the CLI could not reach kenneld, a file could not be read, or kenneld returned an error. |
| 3 | Policy validation or resolution failure: schema, invariants, template inheritance, or include conflict. |
| 6 | Signature or lockfile-pin failure during resolution. |

`compile` and `validate` map `kennel-policy` errors to codes via `policy_error_code`: signature and lockfile-mismatch errors yield 6; parse, canonicalisation, schema-version, invariant, source-validation, resolution, translation, and include-conflict errors yield 3.
