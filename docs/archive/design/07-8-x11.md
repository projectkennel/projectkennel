# §7.8 X11 — out of scope

X11 cannot be granted. It has no useful per-client confinement: any client of an X server can read
every other window's contents (`XCopyArea` against any client's drawables), read every keystroke
(`XGrabKeyboard`, `XQueryKeymap`, the RECORD extension), synthesize input into any window
(`XTestFakeKeyEvent`, `XSendEvent`), read and write the clipboard, screenshot the whole display, and
enumerate every other connected client. XSecurity and Xauth gate *who connects*, not *what a connected
client may do* — there is no protocol mechanism to limit a client once connected. A framework that
confines processes cannot grant X11 and remain honest: a kennel with X11 access holds, as one
categorical capability, "read all user input, see all user windows, control all user windows".

X11 is therefore out of scope:

- There is no `[x11]` config section — an `[x11]` block is an unknown-section error at parse (`02-2`).
- `/tmp/.X11-unix/*` cannot be granted, and the kennel's mount view exposes no host X server socket,
  so there is no path to one from inside a kennel.
- `DISPLAY` is not synthesised into the kennel environment (the env is built from policy, §7.9.2), so
  a workload inherits no display pointer.

Wayland is separate: its per-client capability model admits a mediated grant — the nested-compositor
display service (§7.14), where the workload's display server is a compositor constructed inside a
GUI-service kennel rather than the host's — which X11's shared-display model does not.
