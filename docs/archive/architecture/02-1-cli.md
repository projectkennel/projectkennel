# API surfaces — CLI

## Stability commitment

**Stable** per `02-0-overview.md`. Subcommands, flags, and exit codes are backwards-compatible across minor versions. Human-readable text output is best-effort and may evolve for clarity; scripts must not parse the human format.

Deprecations follow the policy in `02-0-overview.md` §Deprecation policy: announced in CHANGELOG, warned at runtime, minimum one minor version of overlap before removal.

---

## Invocation

```
kennel <SUBCOMMAND> [ARGUMENTS]
```

The CLI is the user's entry point. It is stateless: every invocation starts from disk and either contacts kenneld over its control socket (`run`, `attach`, `stop`, `list`) or operates purely on local files (the `policy` group, `review`, `keygen`, `subkennel`, `audit`). `kennel` does not maintain a long-running connection across invocations — an interactive `run`/`attach` holds its connection only for that terminal session, and detaching ends the connection without ending the kennel.

The CLI dispatches on the first argument; each subcommand parses its own flags. The kenneld control socket is `/run/user/<uid>/kennel/control.sock` (the `kenneld.socket` user unit binds it; kenneld is socket-activated on the first `run`).

### Help

`kennel`, `kennel --help`, `kennel -h`, and `kennel help` all print the command list (one line per top-level verb, rendered from the in-binary command table so it cannot drift from what exists). `kennel <command> --help` prints that command's usage line. The `policy` noun has its own help: `kennel policy` (or `kennel policy --help`/`-h`/`help`) prints the verb list, and `kennel policy <verb> --help` prints that verb's usage.

## Subcommands

The top-level verbs are `run`, `attach`, `review`, `stop`, `list`, `policy`, `keygen`, `subkennel`, and `audit`. `policy` is a noun group with its own sub-verbs — `list`, `show`, `edit`, `generate`, `compile`, `validate`, `sign`, `lint`, `risks`, `diff`, and `upgrade` — covered in `### kennel policy` below.

### `kennel run` — start a workload inside a kennel

```
kennel run <policy> [<name>] [--key <key>] [--force] [--template-dir <dir>]... [--trust-dir <dir>]... [-- <command> [args...]]
```

Asks kenneld to start a kennel from `<policy>` and run a workload inside it. The `--` separator is **optional**:

- **No `--`:** `kennel run <policy>` runs the policy's own embedded `[workload]` (the `argv`/`cwd` carried in the settled policy, `02-2-config-schema.md` §`[workload]`). A policy with no `[workload]` and no `-- <command>` is an error.
- **With `-- <command>`:** the command after `--` overrides the policy's `[workload]`. If the policy **pins** its workload (`pinned = true`), an override is refused unless `--force` is also given. `--force` is only meaningful alongside a `-- <command>`.

The `[workload]` stanza is optional in the settled policy and carries `argv`, `cwd`, the `pinned` flag, and `sha256` — a list of accepted binary digests. When set, **kenneld** verifies the resolved binary's SHA-256 against that list before exec (the pin is a daemon decision made pre-handoff, never in init); an unmatched digest aborts the run.

`<policy>` is resolved either as a literal **file path** or, failing that, as a **policy name** found in the `policies/` cascade (`~/.config/kennel` → `/etc/kennel` → `/usr/lib/kennel`), preferring the settled artefact `policies/<name>/<name>.settled.toml` over the source `policies/<name>/policy.toml` (`07-paths.md` §Run-policy resolution). The settled artefact is enforced as-is and needs **no key** — the daemon verifies its signature against its trust store (system keys plus the user's own `~/.config/kennel/keys`, the trust split). This is the normal flow: `kennel policy compile <name>` once, then `kennel run <name>` keyless. Only when `<policy>` resolves to a **source** leaf is it compiled-and-signed in memory for the run, which needs a signing (private) key — `--key`, or, omitted, the sole key in the user key dir. The private key is never needed merely to run a pre-compiled policy. The templates a settled policy derives from must be system-signed regardless.

`<name>` (the kennel instance) is **optional** and defaults to the resolved policy name — `kennel run ai-coding -- bash` runs `policies/ai-coding/` as a kennel named `ai-coding`.

The kennel runs in the foreground; **kenneld** (not the CLI) performs the spawn sequence. When stdin is **not** a terminal the CLI passes its three stdio descriptors to kenneld over `SCM_RIGHTS` and the workload is attached to them; the CLI blocks until the workload exits and then exits with its exit code (clamped to a byte). When stdin **is** a terminal (e.g. `kennel run interactive -- /bin/bash`) the run is interactive and **detachable**: the CLI passes a single socket instead, the spawn seal allocates a controlling pty inside the kennel's own devpts (so the workload has real job control and the operator's own tty is never exposed), and **kenneld** holds the pty master in a PTY broker that proxies it to this terminal (design §7.9.5a, lifecycle `05-state-and-supervision`). The CLI raw-modes the terminal and proxies bytes both ways. `Ctrl-\ d` **detaches** — the CLI restores the terminal and exits 0 while the workload keeps running (reconnect with `kennel attach <name>`); otherwise the CLI blocks to the workload's exit and exits with its code. Dangerous terminal escapes in the workload's output are filtered **client-side, by the CLI**, on the bytes' way to the real terminal, on by default (`[tty].filter_terminal_escapes`, §02-2); the daemon's broker is a raw-byte router and conveys the policy decision in its `Started`/`Attached` response, keeping the `vte` parser of workload-controlled bytes out of the daemon TCB (§4.8).

Protocol: kenneld first replies `Started { ctx, pid }` (the CLI prints a one-line confirmation to stderr), then `Exited { code }` when the workload terminates (or `Detached { reason }` if another client takes the terminal over). An `Error` at any point is printed to stderr and the CLI exits non-zero. A local `Ctrl-\ d` detach is purely client-side and needs no reply — the CLI restores the terminal and exits 0. Live terminal resizes ride a fire-and-forget `Resize { rows, cols }` request (the broker holds the master).

Each `kennel run` is one kennel: it gets its own context byte, cgroup, egress proxy, loopback addresses, and constructed `$HOME` view, all torn down immediately when the workload exits.

Before contacting kenneld, an interactive or non-interactive `run` also performs trust-manifest **pre-flight** (unless the policy sets `[trust].manifest = false`): it ensures a `.trust-manifest.json` at each writable root so the kennel can mask it (§7.4, T2.8). An existing manifest is left as-is — refreshing pins is `kennel review`, not a `run` side effect.

### `kennel attach` — reattach a terminal to a running kennel

```
kennel attach <name>
```

Reconnects this terminal to a still-running interactive kennel's controlling pty (the docker-`attach`/tmux model; needs a terminal on stdin). kenneld's PTY broker replays the recent output, then **takes over** from any currently-attached client — a single PTY, so the prior terminal receives a clean `Detached { reason }` and exits 0. The same raw-mode proxy and `Ctrl-\ d` detach as an interactive `run` apply; the kennel survives detach. Replies `Attached { ctx, pid }` on success, or `Error` (unknown kennel, not interactive, or already exited) which the CLI prints before exiting non-zero.

### `kennel review` — review and re-pin a workspace trust manifest

```
kennel review <policy> [--yes]
```

Host-side operator sign-off on a workspace's masked trust manifest (`.trust-manifest.json`, §7.4 / T2.8) after legitimate edits. Because the confined workload cannot see or update the manifest (the spawn view masks it), a changed or newly-added execution trigger stays flagged until a human re-pins it here — never contacts kenneld.

Resolves `<policy>` like `run` (preferring the compiled `<name>.settled.toml`; a bare source policy is refused with an instruction to `kennel policy compile` first), reads each writable root's manifest, and prints a diff of **modified** (`~`), **removed** (`-`), and **new, unpinned** (`+`) triggers. For each root with divergences it prompts (`[y/N]`) to re-pin; `--yes` re-pins without prompting, and a non-terminal stdin defaults to *no* (an unattended `review` never silently re-pins). Re-pinning adopts the on-disk hashes and rewrites the manifest, so cooperating host tooling unlocks the workspace. A policy with `[trust].manifest = false`, or one with no manifests yet, reports that and exits 0.

### `kennel stop` — stop a running kennel

```
kennel stop <name>
```

Asks kenneld to stop the named kennel. kenneld terminates the workload and tears down the kennel's per-kennel resources (proxy, loopback addresses, cgroup, constructed view) immediately. Replies `Stopped` on success; the CLI prints a confirmation and exits 0.

### `kennel list` — list running kennels

```
kennel list
```

Asks kenneld for the running kennels and prints a **what-spawned-what tree** (the fleet-topology view): top-level kennels at the root, each ephemeral `SPAWN` sibling nested under the kennel that requested it. The nesting is derived CLI-side from the `spawn-<parent-ctx>-<id>` registry name (§7.12.7) — no daemon-side change — and spawns are depth-1, so the tree is two levels; an orphan spawn whose requester has already torn down renders at the root, flagged `(orphan spawn)`. Each row carries `NAME`, `CTX`, `PID`, `STATE`, `CLIENT`. State is `running` once the workload is launched, otherwise `starting`; `CLIENT` is `attached` when a terminal is currently proxied to the kennel's PTY broker, else `detached` (the kennel runs on regardless). Prints `no running kennels` when the daemon reports none.

### `kennel policy` — author, inspect, and compile policies

`policy` is a noun group with eleven verbs. `kennel policy` (or `--help`/`-h`/`help`) prints the verb list; `kennel policy <verb> --help` prints a verb's usage. Every verb is stateless — none contacts the daemon.

| Verb | Purpose |
|---|---|
| `list` | enumerate policies and templates in the search path |
| `show` | print what a policy resolves to (the effective policy) |
| `edit` | open a policy's source in `$EDITOR` |
| `generate` | scaffold a new leaf policy |
| `compile` | resolve a source policy into a signed settled artefact |
| `validate` | resolve and check a policy without writing an artefact |
| `sign` | sign a source template/fragment with a key |
| `lint` | check the shipped template corpus for incoherences |
| `risks` | evaluate a policy against the threat catalogue (exposures, residuals) |
| `diff` | interpreted grant delta between a policy and its baseline (or another policy) |
| `upgrade` | re-pin a policy's template to a newer version (with review) |

#### `kennel policy list` — enumerate policies and templates

```
kennel policy list
```

Walks the `policies/` and `templates/` cascades (`~/.config/kennel` → `/etc/kennel` → `/usr/lib/kennel`) and prints each artefact's name, its kind, and the directory it was found in. A directory is reported as `settled` if it holds a compiled `<name>.settled.toml`, else its `policy.toml` is classified `template` (carries `template_name`), `leaf` (carries `name`), or `source` (neither/unparseable). A closer directory's entry shadows a farther one. A read-only survey; prints a note if the search path is empty.

#### `kennel policy show` — show what a policy resolves to

```
kennel policy show <policy> [--template-dir <dir>]... [--trust-dir <dir>]...
```

Resolves `<policy>` — compiling a source policy in memory, or verifying-and-reading a settled artefact — and prints the **effective policy** in human-readable form: the network posture (the `net.mode` tier and whether an egress proxy stands up), the filesystem read/write grants, the exec allowlist size, the embedded `[workload]` (its `argv`, whether it is pinned, and any SHA-256 pins), and the TTL. This is the "what does this template actually resolve to" tool — it catches a template that *says* one thing but *resolves* to another (e.g. an `open` policy that still carries proxy-constrained egress). `--template-dir` and `--trust-dir` behave as in `compile`.

#### `kennel policy edit` — edit a policy's source

```
kennel policy edit <name>
```

Resolves `<name>` to its source `policy.toml` and opens it in `$EDITOR` (then `$VISUAL`, else `vi`). If the source lives in a read-only system dir (`/etc/kennel`, `/usr/lib/kennel`), it is **copied into the user's `~/.config/kennel/policies/<name>/` first** (copy-on-write) so edits never try to mutate the system copy; the user copy then shadows the system one in the cascade. An existing user copy is edited in place.

#### `kennel policy generate` — scaffold a new leaf policy

```
kennel policy generate <name> [--from <template>]
```

Scaffolds a minimal leaf policy at `~/.config/kennel/policies/<name>/policy.toml` that inherits `--from` (default `base-confined@v1`), with a commented `[workload]` stub to fill in. `--from` must be a versioned reference (`<template>@v<n>`). Refuses to overwrite an existing policy. Prints the next steps (`kennel policy show <name>`, then `kennel policy compile <name>`).

#### `kennel policy compile` — resolve a source policy into a settled policy

```
kennel policy compile <policy> [--output <path>] [--key <key> | --unsigned]
                      [--require-signed] [--no-lock] [--template-dir <dir>]... [--trust-dir <dir>]...
```

Resolves a source policy fully — template-chain fold, signed includes, leaf deltas, source-signature verification, lockfile byte-checks, invariant validation, and installation-constant substitution — and writes a **settled policy**: a flat, canonical-TOML artefact the runtime consumes directly. See `02-2-config-schema.md` §The settled policy and design doc §9.10. Stateless: `compile` never contacts the daemon.

`<policy>` is a path or a **policy name** resolved from the `policies/` cascade, preferring the source `policy.toml` it is about to compile. By default the settled artefact and lockfile are written back into the same `policies/<name>/` folder, so `kennel policy compile <name>` then `kennel run <name>` is the normal author loop.

The settled policy is **signed by default** — with `--key`, or, omitted, the sole signing key in the user key dir (`~/.config/kennel/keys`; ambiguous if there are several, none if there are zero — pass `--key` then). `--unsigned` opts out, sealing an unsigned development build (`algorithm = "none"`, content-complete). `--key` and `--unsigned` are mutually exclusive.

Flags:

- `--output <path>` — where to write the settled policy. Default: `<policy-dir>/<name>.settled.toml`.
- `--key <path>` — ed25519 signing (private) key for the settled policy. Omitted, the sole key in the user key dir is used.
- `--unsigned` — emit a content-sealed but unsigned settled policy (development use).
- `--require-signed` — refuse unsigned templates during resolution; verify every template against the trust store.
- `--no-lock` — skip writing/verifying the `<name>.lock` file beside the output.
- `--template-dir <dir>` — additional template search path (repeatable). The user (`~/.config/kennel/templates`) and system (`/etc/kennel/templates`) directories are always searched.
- `--trust-dir <dir>` — additional **template**-trust path holding `<key_id>.pub` keys (repeatable). Templates verify against the **system** stores only (`/etc/kennel/keys`, `/usr/lib/kennel/keys`) — never `~/.config/kennel/keys` (the trust split); `--trust-dir` lets an operator add an org key dir.

A `<name>.lock` file beside the output byte-pins the resolved references. A re-tagged or re-signed reference relative to a prior lockfile is an integrity failure (exit 6).

Exit codes: 3 for policy-validation or resolution failure, 6 for a source-signature or lockfile-pin failure.

Resolution settles `net.mode` to one of the four tiers — `none`, `constrained`, `unconstrained`, `open` (`02-2-config-schema.md` §`[net]`) — and runs the mode's invariant checks. Coherence findings that contradict the resolved mode (e.g. proxy-constrained egress carried by an `open` policy) are surfaced by `kennel policy lint` (below).

#### `kennel policy validate` — validate a policy without emitting an artefact

```
kennel policy validate <policy> [--require-signed] [--template-dir <dir>]... [--trust-dir <dir>]...
```

Resolves and checks a policy (chain, signatures, deltas, includes, invariants) without writing a settled policy. Exits 0 if it resolves cleanly (printing a one-line summary with the reference and deferred-substitution counts); otherwise the same exit code `compile` would return for that failure. Flags `--require-signed`, `--template-dir`, and `--trust-dir` behave as in `compile`, and `validate` runs the same invariant checks.

#### `kennel policy sign` — sign a source template or fragment

```
kennel policy sign <template> --key <key> [--output <path>]
```

Signs a source template or fragment with an ed25519 key, **appending** a `[signature]` block to the file (the signature covers the canonical re-serialisation, not the raw bytes, so comments and formatting are preserved). Refuses a file that already carries a `[signature]`. Leaf policies need not be signed.

Prints the public key to install in the trust store as `<key_id>.pub`.

Flags:

- `--key <path>` — required. The ed25519 signing key.
- `--output <path>` — write the signed file here instead of overwriting the input.

#### `kennel policy lint` — check the template corpus for incoherences

```
kennel policy lint [--template-dir <dir>]... [--trust-dir <dir>]...
```

Compiles every `<name>/policy.toml` found in the template cascade (in memory, development trust) and runs the coherence lint on each resolved policy, reporting findings — settings that contradict the resolved `net.mode`, or grants the mode makes vacuous. Each finding is printed as `<template>: <finding>`; a template that fails to compile is reported too. Exits **0** when every template lints clean, **7** if any template produces a finding (a CI-friendly code distinct from the resolution/signature codes). `--template-dir` and `--trust-dir` behave as in `compile`.

#### `kennel policy risks` — evaluate a policy against the threat catalogue

```
kennel policy risks <policy> [--template-dir <dir>]... [--trust-dir <dir>]... [--json]
```

Resolves the policy (chain folded, like `validate`) and reports what its grants **expose** and **mitigate** against the threat framework, each with the granting site, its documented `reason`, and the catalogue residual. It is **source-driven**: the `threats`/`reason` tags live only in the source policy and the compiler's derivation, never the settled artefact, so `risks` reads the resolved *source*.

For each threat-carrying grant (`[[net.proxy.allow]]`, `[[net.proxy.deny]]`, `[[net.bpf.*]]`, `[[unix.allow]]`, `[[fs.dev.passthrough]]`, `[[ssh.destinations]]`, `[[binder.*]]`) it surfaces the authored `threats.exposed`/`threats.mitigated` tags, and it adds the compiler-**derived** exposures (`mode = host` → T1.6, a device passthrough → T2.1, `[ssh].allow_headless` → T1.6), marked `(derived)` so an inferred tag is distinguishable from an authored one. A tag that names no catalogued threat is flagged as a likely typo rather than dropped.

The threat catalogue is `docs/reference/THREATS.md` (canonical prose) and its machine form `dist/threats/catalogue.toml` (the id/title/residual table this command maps against, kept in sync by a CI check). The CLI reads the installed catalogue (`/etc/kennel` then `/usr/lib/kennel`), falling back to a copy embedded in the binary, so the command works from a source checkout. `--json` emits the structured report for CI/tooling. Read-only; exits **0** (the report is informational — a future `--strict` could make unknown/missing-required tags non-zero). The fold accepts either policy form — a template/source document or a delta-leaf (`[[fs.read.add]]`, …) — via the shared `effective_source`, so `risks` works on a leaf policy as well as a template.

#### `kennel policy diff` — interpreted grant delta between two effective policies

```
kennel policy diff <policy> [<other>] [--template-dir <dir>]... [--trust-dir <dir>]... [--json]
```

The semantic counterpart of `policy upgrade`'s raw source line diff (`05-templates.md` §5.11/§5.13): instead of added/removed *text lines*, `diff` reports added (`+`), removed (`-`), and modified (`~`) **grants** — which capabilities widened or narrowed — each annotated with the threats it exposes/mitigates and a net **threat-posture delta**. Like `risks` it is **source-driven** (the same folded effective source, threats intact) and accepts either policy form on each side.

- **One argument** diffs the policy against its **template baseline** — the template named in its `template_base`, resolved with none of the leaf's own deltas. This is the "what does my policy add over the template it inherits" view (the §5.13 *your-deltas* report). A policy with no `template_base` (a root template) is refused with an instruction to pass a second policy.
- **Two arguments** diff `<policy>` → `<other>`: `<policy>` is the *before*, `<other>` the *after*. Use it to review a user policy against an organisational baseline, or one template version against another (compile-free, complementing `upgrade`'s text diff).

Each change carries the granting site, its documented `reason`, the catalogued threats it `exposes`/`mitigates` (with one-line residuals), a `(widens reach)` marker when the change loosens confinement, and an honest impact note for the unambiguous cases (a removed deny weakens the floor; a permissive `exec.allow` runs anything). The capability surface compared spans `exec`, `fs.read`/`fs.write`/`fs.dev.passthrough`, the `net` posture (`mode`, `[[net.proxy.*]]`, `[[net.bpf.*]]`), `[[unix.allow]]`, `[ssh]`, `[[binder.*]]`, `[identity].groups`, the `[workload]` pins, and the `[lifecycle]`/`[tty]`/`[trust]` toggles — the grants that shape confinement; audit-sink and env-curation knobs are omitted. The net threat-posture delta reuses the `risks` engine on each side. Terminal output routes every policy-sourced string through `sanitise_for_log` (§10.4); `--json` emits the delta through `serde_json` (a real serialiser, §10.3). Read-only; never contacts the daemon; exits **0**.

### `kennel keygen` — generate an ed25519 signing key

```
kennel keygen <key-id> [--dir <dir>] [--force]
```

Generates an ed25519 keypair from the OS CSPRNG and writes it into the user key
dir (default `$XDG_CONFIG_HOME/kennel/keys`, else `~/.config/kennel/keys`):
`<key-id>.key` is the base64 seed (mode `0600`, the signing key) and
`<key-id>.pub` is the base64 public key. The `<key-id>` is both the filename and
the signature `key_id` (restricted to 1–64 chars of `[A-Za-z0-9._-]`). Refuses to
overwrite an existing seed without `--force` — replacing a key invalidates
everything signed with the old one.

The daemon trusts the user's own `~/.config/kennel/keys` for settled **run
policies** out of the box (the trust split, `07-paths.md` §Policy-signing trust
split), so no further setup is needed: `kennel policy compile <name>` signs the policy
with this key (picked automatically when it is the only one), and `kennel run
<name>` then runs it **keyless** — the private seed is only touched at compile/sign
time, never to run. The key **cannot** sign templates; those verify against system
keys only. Only to let *other* users or a fleet trust your signatures — or to sign
templates — `keygen` prints the command to copy the *public* key into the system
trust store (`/etc/kennel/keys/<key-id>.pub`). Stateless: `keygen` never contacts
the daemon.

Flags:

- `--dir <dir>` — write the keypair here instead of the default user key dir.
- `--force` — replace an existing `<key-id>.key` (and its `.pub`).

### `kennel subkennel` — provision/validate the per-user allocation

```
kennel subkennel add   [--uid N] [--namespace NS] [--tag N] [--file PATH]
kennel subkennel check [--uid N] [--file PATH]
```

The `/etc/kennel/subkennel` allocation (`uid:tag:gid:namespace`, §07-paths) is what
lets a user's kenneld start; a missing or malformed line stops it cold. `add`
generates a provably-valid line for a user (default: the current uid) — the lowest
free 12-bit `tag` (from 1; tag 0 is reserved, its `/20` contains `127.0.0.1`) and a
fresh random 40-bit ULA `gid`, both checked against existing lines so two users
never collide. The **namespace defaults to `kennel-<user>`**; `--namespace` only
overrides it. The file is root-owned, so `add` prints the line plus the exact `sudo`
command to append it (it never writes the file itself), then re-derives nothing. `check`
parses the file, reports malformed lines and duplicate `uid`/`tag`/`gid`, and exits
non-zero if the named user has no valid allocation — the precondition kenneld needs.
Stateless: neither contacts the daemon.

Flags:

- `--uid <N>` — operate on uid `N` (default: the caller's real uid).
- `--namespace <NS>` — override the `kennel-<user>` default (`add`).
- `--tag <N>` — use this tag instead of the lowest free one (`add`).
- `--file <PATH>` — a different allocation file (default `/etc/kennel/subkennel`).

### `kennel audit` — query a kennel's audit log

```
kennel audit <name> [--resource CLASS] [--since DUR] [--novel-only] [--follow] [--print-journalctl-command]
```

Reads the per-kennel JSONL audit files directly from the file sink
(`$XDG_STATE_HOME/kennel/<name>/<class>.jsonl`, falling back to
`~/.local/state/kennel/…`, §02-3) — no daemon round-trip, queryable from a fresh
shell. Events from the selected class files are printed in `ts` order.

Flags:

- `--resource <class>` — limit to one class: `net` (→ `network.jsonl`), `fs`
  (→ `filesystem.jsonl`), `exec`, `unix`, `dbus`, `priv`, or `lifecycle`. Default:
  all classes present.
- `--since <dur>` — keep only events newer than the given age (`90s`, `30m`, `2h`,
  `7d`; a bare number is seconds). Compared against each line's RFC3339-UTC `ts`.
- `--novel-only` — collapse repeats: lines identical but for their timestamp are
  shown once (their first occurrence), so a flood of identical denials reads as one.
- `--follow` — after the initial dump, stream new events as they are appended
  (re-reading from the start if a file was rotated).
- `--print-journalctl-command` — instead of reading files, print the equivalent
  `journalctl --user KENNEL_KENNEL=<name> [KENNEL_RESOURCE=<class>] [--since "<dur> ago"]`
  invocation, for journald-only deployments.

A `<name>` containing `/` or `..` is refused (it is a path component). A kennel with
no audit files yet prints a note and exits 0.

#### `kennel policy upgrade` — re-pin a policy's template to a newer version

```
kennel policy upgrade <name> [--yes] [--template-dir D]... [--trust-dir D]...
```

Detects whether the policy's `template_base` has a newer version available in the
template search path. If so, it prints the **source diff** between the pinned and
the new template version, asks for consent, and on yes rewrites the leaf's
`template_base` and recompiles so `kennel.lock` re-pins to the new bytes. This is
the sanctioned way to change a locked entry (§The lockfile, `02-2`): the lock is
otherwise immutable and a mismatch is a hard error. Project Kennel never
auto-upgrades — `kennel run` warns when a newer version exists but keeps running
the pinned one.

Flags:

- `--yes` / `-y` — migrate without the interactive confirmation (scripts/CI). With
  no terminal and no `--yes`, the command refuses rather than migrate unattended.
- `--template-dir <D>` — add to the template search path (repeatable); forwarded to
  the recompile.
- `--trust-dir <D>` — the trust store the new version's signature verifies against;
  forwarded to the recompile.

The richer semantic threat-impact delta (the annotated `+/~/-` view in
`05-templates.md` §5.11) is `kennel policy diff`'s job (above); `upgrade` shows the
honest *source* line diff, and `kennel policy show` prints the full effective
policy after upgrading.

## Exit codes

| Code | Meaning |
|---|---|
| 0 | Success. For `kennel run`, the workload's own exit code is passed through (clamped to a byte). |
| 1 | Generic failure: the CLI could not reach kenneld, a file could not be read, or kenneld returned an error. |
| 3 | Policy validation or resolution failure: schema, invariants, template inheritance, or include conflict. |
| 6 | Signature or lockfile-pin failure during resolution. |
| 7 | `kennel policy lint` found at least one coherence finding across the template corpus. |

`kennel policy compile` and `kennel policy validate` map `kennel-lib-policy` errors to codes via `policy_error_code`: signature and lockfile-mismatch errors yield 6; parse, canonicalisation, schema-version, invariant, source-validation, resolution, translation, and include-conflict errors yield 3. `kennel policy lint` is the only command that returns 7 (a distinct, CI-friendly code for a corpus with findings).
