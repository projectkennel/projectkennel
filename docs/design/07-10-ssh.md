# ¬ß7.10 Policy surface: per-kennel SSH egress

SSH is a capability domain of its own. A kennel that does real work ‚Äî an AI
coding agent, a build ‚Äî needs to `git push`, `git pull`, or `ssh` to a small set
of hosts, using *selected* keys. Project Kennel grants this through the dedicated
`[ssh]` section and the ¬ß7.10 re-origination bastion: synthetic keys bound to
forced commands, one per destination. The workload holds no real key, holds no
agent socket, and cannot choose its own destination.

## 7.10.1 Why not an agent socket: the destination-blind oracle

The ssh-agent wire protocol carries an opaque to-be-signed blob, **not a
hostname**. An agent ‚Äî or a fingerprint-filtering broker in front of one ‚Äî can
therefore bound *which key* signs but never *which host* the signature
authenticates to. A hostile workload that holds the socket opens it directly
(ignoring the stock client and any curated `~/.ssh/config`), requests a
signature for an allowlisted key over a challenge it crafted for an
attacker-controlled host, and authenticates as the user anywhere that key is
accepted ‚Äî cross-host key reuse. Fingerprint filtering and a curated config
constrain only the client the workload is free to bypass. The agent is a
*destination-blind signing oracle*; see the T1.6 residual in `THREATS.md`.

Per-kennel SSH is routed through a **re-origination bastion** instead. The
workload holds no real key, holds no agent socket, and cannot choose its own
destination.

## 7.10.2 Topology

A single per-user **`kennel-sshd`** ‚Äî a managed instance of stock OpenSSH
`sshd` ‚Äî runs alongside `kenneld` for the session, on a loopback port. It holds
no keys; it is a forced-command router whose lifecycle and key state `kenneld`
owns. It is reached over the egress proxy (¬ß7.10.4), so it sits on the network
boundary, not the AF_UNIX boundary of ¬ß7.6.

## 7.10.3 The synthetic key is the destination selector

For each **destination** a kennel is granted (the `[ssh]` policy, ¬ß7.10.8),
`kenneld` mints ‚Äî at **compile time** ‚Äî a **synthetic** ed25519 keypair: the
private half goes into the kennel's constructed `~/.ssh`; the public half is
recorded in the signed settled artefact and bound, via the bastion's
`AuthorizedKeysCommand` (‚Üí `kenneld`), to a forced command that bakes in the
destination and the operator-supplied host-side `ssh` options:

```
restrict,pty,command="ssh <options‚Ä¶> -- git@github.com \"$SSH_ORIGINAL_COMMAND\"" <synthetic-pub>
```

The destination ‚Äî and which real key/port/config the host-side `ssh` uses (the
`options`, e.g. `-i ~/.ssh/id_x`) ‚Äî is fixed by *which synthetic key
authenticated*, never parsed from anything the workload sends. A workload holding
`synthetic-github` can only ever reach github; it cannot redirect the forced
command, and a non-synthetic key is refused. There is no oracle: each synthetic
key is a capability for exactly one destination. The keypair is persisted beside
the artefact (so the kennel's `~/.ssh` is stable) and its public half is pinned
by the policy signature ‚Äî the bastion authorises only a key the signature covers.

## 7.10.4 Re-origination

On connection the bastion's forced command is simply `ssh <options> -- <dest>
"$SSH_ORIGINAL_COMMAND"`, run **as the operator** ‚Äî no privilege, no helper
binary. The operator's per-destination `options` name which real key the outbound
`ssh` signs with, from the operator's **own host-side store** (`~/.ssh`); Project
Kennel stores no key material and there is **no agent in the path** (an exposed
agent socket would be the destination-blind oracle of ¬ß7.10.1).
`$SSH_ORIGINAL_COMMAND` is forwarded as a single argument, so `git`-over-ssh and
interactive shells both work, and the real key signs against a destination the
bastion fixed ‚Äî so cross-host reuse is structurally impossible.

The bastion is reached over the existing per-kennel egress proxy (¬ß7.5): the
host-side `host-netproxy` delegate dials its loopback endpoint on the kennel's
behalf ‚Äî exactly as it dials any egress target for the SOCKS facade ‚Äî so there is
no SSH-specific transport. The kennel's `ssh` reaches it through `ProxyCommand
facade-ssh`, an `INet` `CONNECT_INET` transaction to kenneld over binder, which
returns the connection fd. The destination itself is reached **host-side by the
forced command's `ssh`, outside the kennel's egress**: the kennel allowlists only
the bastion endpoint (a host-service literal), never the destination.

## 7.10.5 The synthetic `~/.ssh`

The kennel's constructed `~/.ssh` (a tmpfs subtree of the constructed `$HOME`,
¬ß7.4.5, mode `0700`) holds only the disposable synthetic private key(s); a
generated, read-only `config` ‚Äî one stanza per granted host, all
`HostName`‚Üíbastion, `HostKeyAlias kennel-bastion`, `IdentityFile`‚Üíthe matching
synthetic key, `IdentitiesOnly yes`, `StrictHostKeyChecking yes`; and a
`known_hosts` carrying only the bastion's host key (under `kennel-bastion`).
Everything real is structurally absent (ENOENT): the user's keys, real `config`,
real `known_hosts`, other kennels' material. The generated config leaks nothing
‚Äî it is derived entirely from already-granted policy.

## 7.10.6 Lockdown and scope

`kennel-sshd`'s config and the per-key `restrict,pty` option deny everything but
the forced command and a pty ‚Äî `AllowTcpForwarding no`, `X11Forwarding no`,
`AllowAgentForwarding no`, `Subsystem sftp /bin/false`. **SFTP, scp, and
port-forwarding are out of scope for the first cut and denied; revisit later.**
Per-signature gating is delegated to the user's key custody: a hardware/sk key
gives touch-per-use for free; a non-interactive key is usable freely *for its
granted destinations only* ‚Äî there is no Project-Kennel prompt to bypass, so the
`[ssh] allow_headless` flag (¬ß7.10.8) governs whether a non-interactive kennel
may drive such a key at all.

## 7.10.7 Where keys and trust live

- Real private keys live only in the user's host-side store; the forced command's
  `ssh` (run as the operator) uses them by the policy's per-destination `options`
  (`-i ‚Ä¶`), and the kennel never sees them. Project Kennel mints and holds only the
  disposable **synthetic** capability keys.
- The bastion's `AuthorizedKeysCommand` helper is **root-owned** (installed in
  the prefix) and queries `kenneld` over its control socket; the rootless
  `kennel-sshd` only invokes it ‚Äî OpenSSH's safe-path check rejects an AKC that
  is not root-owned.
- The bastion's hardened `sshd_config` is **surfaced as a template** in the root-owned
  cascade ‚Äî `/etc/kennel/kennel-sshd.conf` (admin override) over the vendor
  `/usr/lib/kennel/kennel-sshd.conf`, with a byte-identical copy compiled into `kenneld`
  as the fallback ([[no-hardcoded-paths-config-cascade]]). The lockdown posture (¬ß7.10.6 ‚Äî
  publickey-only, no forwarding/agent/tunnel, `PermitOpen none`, SFTP‚Üí`/bin/false`) lives
  in that file, **legible and admin-tunable**, not baked into the daemon. There is **no
  user/XDG layer**: the bastion lockdown must not be weakenable by an unprivileged
  workload (which cannot write `/etc/kennel` or `/usr/lib/kennel`). `kenneld` resolves the
  template, substitutes the per-bastion values (`ListenAddress`/`Port`/`HostKey`/`PidFile`
  and the authorised-keys source), and writes the rendered config to a `0700`
  `$XDG_RUNTIME_DIR` runtime dir alongside the generated host key, gone at teardown.
- The host-side `known_hosts` the forced command verifies against is the
  operator's own (or one named by their `options`, e.g.
  `-o UserKnownHostsFile=‚Ä¶`); a destination with no known host key fails closed at
  `StrictHostKeyChecking`. This is the operator's host-side configuration, not
  anything the kennel supplies.
- **Hardware tokens** (FIDO/sk keys) are used *host-side* by the forced command's
  `ssh`, so the touch happens on the host. SSH must **not** grant `fido_token` /
  bind `/dev/hidraw*` into the kennel ‚Äî that would let the workload drive the
  authenticator (CTAP) directly, bypassing the touch defence (¬ß7.9.10).

## 7.10.8 `[ssh]` policy (source-only)

A compile-time-only section, resolved and folded like `[unix]` and dropped from
the settled `EffectivePolicy`:

```toml
[ssh]
# Whether a granted destination may be used by a non-interactive (CI) kennel with
# no per-use touch/confirmation. Loud, threat-tagged; default false.
allow_headless = false

[[ssh.destinations]]
dest    = "git@github.com"      # the destination the host-side `ssh` connects to
options = ["-i", "~/.ssh/id_github", "-o", "IdentitiesOnly=yes"]  # host-side ssh opts
reason  = "push to the project's github remote"
threats = { exposed = ["T1.6"] }
```

There is **no real-key fingerprint and no agent**: the operator names *which* real
key (and port/config) the host-side `ssh` uses per destination via `options`,
which run as the operator in their own key store. `kenneld` mints one synthetic
capability keypair per destination at compile time.

The destination is **not** a kennel egress target ‚Äî it is reached host-side by the
forced command, outside the kennel's allowlist (only the bastion endpoint is
granted, as a host-service literal). So there is no `dest ‚äÜ net.allow` check.

Compile-time validation: every `dest` is non-empty; `allow_headless = true` is
loud and carries a threat tag.

## 7.10.9 Test plan additions

A regression test for each invariant of this section:

1. **No-grant route absence.** A kennel with no `[ssh]` grant has no route to the bastion and no synthetic `~/.ssh`.
2. **Re-origination and command forwarding.** Through a real bastion `sshd` + destination `sshd` (stock OpenSSH, no agent), a granted destination is reached over the `ssh` ‚Üí connector ‚Üí proxy ‚Üí bastion chain, with the forced command (`ssh <options> -- <dest>`, run as the operator) rewriting the origin and forwarding `$SSH_ORIGINAL_COMMAND`. Proven end to end by the `ssh-egress` policy-suite case (`src/tools/policy-e2e.sh`).
3. **Production key source.** The root-owned `AuthorizedKeysCommand` authorises a synthetic key bound to a live edge and refuses an unregistered key.
4. **Synthetic `~/.ssh`.** A confined kennel with an `[ssh]` grant sees the generated `config`/`known_hosts` and the synthetic key ‚Äî nothing real ‚Äî materialised in the sandbox.
5. **Two-kennel isolation.** Two kennels with different grants each reach only their own granted destinations.
6. **Adversarial denies.** A key used against a non-granted host, and a non-interactive use of a non-`allow_headless` key, are refused.
7. **Compile-time validators.** The ¬ß7.10.8 validators reject a `[[ssh.destinations]]` entry with no `dest`, and require a threat tag when `allow_headless = true`.

The invariants, each with a regression test:

1. A kennel with no `[ssh]` grant has no synthetic `~/.ssh` and no SSH route: `ssh` to any host fails (the bastion endpoint is not allowlisted, and direct `:22` is denied by the egress allowlist).
2. A kennel granted `git@github.com` runs `ssh -T git@github.com` / `git ls-remote`; expect the bastion to re-originate with the operator's real key (named by the destination's `options`). A workload that opens the bastion connection by hand and asks for a *different* destination cannot redirect it ‚Äî the destination is fixed in the forced command, keyed to the synthetic key it authenticated with.
3. A non-synthetic key offered to the bastion is refused.
4. Read `~/.ssh/`: expect only the synthetic private key(s) and the generated read-only `config`/`known_hosts` ‚Äî never the user's real keys, real `config`, or real `known_hosts`.
5. Two kennels granted different hosts each reach only their own granted host; a synthetic key minted for one cannot reach the other's host (each is bound by its own forced command).
6. `restrict,pty` + the `Match` block: a port-forward / SFTP / agent-forward request is refused.
7. A `[[ssh.destinations]]` entry with no `dest` fails to compile; `allow_headless = true` carries the threat tag.
