# §7.8 Policy surface: X11 — non-goal

**X11 is a non-goal.** It cannot be granted, and the per-kennel isolated-X11 paths once sketched
here (Xwayland/Xephyr) are **descoped** — removed from the roadmap, not deferred. The primary
workloads (AI coding agents and headless tools) need no display, and a per-kennel X server is
significant surface to carry for a capability nothing in scope uses. This section records *why X11
cannot be granted* (the standing security position, enforced) and that the isolated path is out of
scope (not built, not planned).

## 7.8.1 Why X11 cannot be granted

X11 has no useful per-client confinement. Any X11 client can:

- Read every other window's contents (`XCopyArea` on the root window or against any other client's
  drawables).
- Read every keystroke going to any other window (`XGrabKeyboard`, `XQueryKeymap`, the RECORD
  extension).
- Synthesize input into any other window (`XTestFakeKeyEvent`, `XSendEvent`).
- Read and write the clipboard arbitrarily.
- Take screenshots of the whole display.
- Enumerate every other client connected to the server.

There is no protocol mechanism to prevent any of this. The X server treats every connected client as
fully trusted with respect to every other client; XSecurity and Xauth gate *who can connect*, not
*what they can do once connected*. A framework that claims to confine processes cannot grant X11 and
remain honest: any kennel with X11 access has, as a categorical capability, "read all user input, see
all user windows, control all user windows".

## 7.8.2 What is enforced

- Granting the host X socket directly (`/tmp/.X11-unix/*`) is **rejected** — it is an ordinary
  `fs` grant to a host path the policy must name, and the per-kennel mount view does not expose the
  host display socket, so there is no path to it from inside a kennel regardless.
- `DISPLAY` is not synthesised into the kennel environment (the env is built from policy, §7.9.2),
  so a workload inherits no display pointer.
- There is **no `[x11]` config section** — the `xwayland_isolated`/`xephyr_isolated` surface is not
  in the schema; an `[x11]` block is an unknown-section error at parse (`02-2`).

## 7.8.3 The descoped path (out of scope)

The considered design was a dedicated X server per kennel (Xwayland on a Wayland host, Xephyr on an
X11 host), so a kennel's clients could only reach each other, never the user's session. It is
recorded here as **a non-goal**: should a display-bearing workload class ever come into scope, this
is the shape to revisit — a separate per-kennel X server, never a bind of the host socket — but it
is not on the roadmap and nothing in the product depends on it.
