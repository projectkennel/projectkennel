# §7.8 Policy surface: X11 (isolated only)

X11 cannot be granted directly. Granting `/tmp/.X11-unix/X0` gives the kennel full input, clipboard, screen, and window control over the user's session, with no protocol mechanism to limit it. Project Kennel rejects policies that attempt this.

For X11-requiring workloads, two isolated paths are provided. Both spawn a dedicated X server per kennel, so the kennel's clients can only interact with each other, never with the user's main session.

## 7.8.1 Direct X11 access

X11 has no useful per-client confinement. Any X11 client can:

- Read every other window's contents (XCopyArea on the root window or against any other client's drawables).
- Read every keystroke going to any other window (XGrabKeyboard, XQueryKeymap, the RECORD extension).
- Synthesize input into any other window (XTestFakeKeyEvent, XSendEvent).
- Read and write the clipboard arbitrarily.
- Take screenshots of the whole display.
- Enumerate every other client connected to the server.

There is no protocol mechanism to prevent any of this. The X server treats every connected client as fully trusted with respect to every other client. XSecurity and Xauth gate *who can connect*, not *what they can do once connected*.

A framework that claims to confine processes cannot grant X11 and remain honest. Any kennel with X11 access has, as a categorical capability, "read all user input, see all user windows, control all user windows".

Project Kennel therefore rejects any policy that would grant `/tmp/.X11-unix/*` directly. The schema validator refuses to load such policies. The `DISPLAY` env var is stripped on kennel entry unless the isolated-X11 path is enabled.

## 7.8.2 Two isolated paths

| Path | Host display server | Per-kennel X server | Maturity |
|---|---|---|---|
| **Xwayland-isolated** | Wayland | Dedicated Xwayland instance | Modern, well-supported |
| **Xephyr-isolated** | X11 | Dedicated Xephyr instance | Old, stable, well-trodden |

Project Kennel picks automatically based on the host environment: Xwayland-isolated on Wayland hosts, Xephyr-isolated on X11 hosts. Templates declare "this kennel needs X11"; Project Kennel selects the path.

Both paths are real X servers from the kennel's perspective. Applications work without modification. The kennel's clients see a complete X server; they just cannot see anything outside their own isolated X server.

## 7.8.3 Xwayland-isolated

On a Wayland host, Project Kennel launches a dedicated Xwayland instance per kennel. Xwayland is itself a Wayland client; the host Wayland compositor sees Xwayland as a single connected client and applies Wayland's per-client capability model to it. The X11 applications inside the Xwayland instance can see and control each other (because they're all clients of the same Xwayland) but cannot reach anything outside the Xwayland instance.

Setup:

```
1. Launch Xwayland with a unique display number (e.g. :42)
   and a per-kennel Xauthority file
2. Wait for Xwayland to be ready (poll or wait on its readypipe)
3. Bind-mount Xwayland's socket into the kennel's shim:
   /tmp/.X11-unix/X42 → /run/kennel/<ctx>/x11/X0
   inside the kennel: /tmp/.X11-unix/X0
4. Set DISPLAY=:0 and XAUTHORITY=<per-kennel-xauth> in kennel env
5. Context starts; X11 apps connect to "X0" which is really Xwayland's :42
```

Xwayland in this mode behaves as a single Wayland surface (or set of surfaces) to the host compositor. The user sees the kennel's windows in their normal Wayland session; the compositor places them, decorates them, focuses them per its policy. Input flows from the compositor to Xwayland based on focus; the kennel's clients receive input only when their windows are focused.

## 7.8.4 Xephyr-isolated

On an X11 host, Project Kennel launches a dedicated Xephyr instance per kennel. Xephyr is a Kdrive-based X server that displays itself as a window inside another X server. The host runs an X server (`:0`); Xephyr runs as a regular X11 client on `:0`, opens a window, and inside that window is a complete X server (`:42`). Clients started with `DISPLAY=:42` connect to Xephyr, not to the host's X server.

Setup:

```
1. Allocate display number :42 (next free)
2. Generate Xauthority cookie for :42
3. Launch Xephyr in host's X session:
   Xephyr -auth <ctx-xauth-file> -screen <geometry> -no-host-grab :42
4. Bind-mount Xephyr's X socket into kennel's shim:
   /tmp/.X11-unix/X42 → /run/kennel/<ctx>/x11/X0
   inside kennel: /tmp/.X11-unix/X0
5. Bind-mount per-kennel xauth file
6. Set DISPLAY=:0, XAUTHORITY=<per-kennel-xauth>
7. Context starts
```

`-no-host-grab` is essential. Without it, Xephyr can grab the host's keyboard when its window has focus and refuse to release. Real DoS vector against the user's main session. Project Kennel must set this flag.

The X11 clients inside Xephyr can do all the usual X11-malicious things — keylog, screen-grab, input-synthesize, clipboard-read — but *only against other clients connected to Xephyr*. They cannot reach `:0`. The host's X server is a separate process with its own socket; the inside-clients have no access to it.

## 7.8.5 Policy primitives

```toml
[x11]
# X11 cannot be set "enabled = true" at the top level. The schema rejects this.
# To use X11, set one of the isolated paths:

xwayland_isolated = false       # requires host Wayland
xephyr_isolated = false         # requires host X11

# At most one of these may be true. If neither is set, no X11 access.

# Display geometry for the isolated server
geometry = "1280x800"
depth = 24

# Whether to bridge clipboard between kennel's X server and the host.
# Default: false. Bridging is a real exfiltration channel.
clipboard.bridge = false
clipboard.bridge_direction = "none"     # "none" | "to-host" | "from-host" | "bidirectional"

# Whether the user can resize the kennel's display window.
# Affects only Xephyr; Xwayland-isolated geometry is compositor-managed.
resizable = true

# Whether to spawn a window manager inside the isolated server.
# Without one, a single fullscreen app works fine; multiple apps need a WM.
window_manager = "openbox"      # "" | "openbox" | "i3" | "fluxbox" | <other>
```

Schema validation:

- `xwayland_isolated` and `xephyr_isolated` cannot both be true.
- Wayland host + `xephyr_isolated = true`: warn (Xephyr works under Xwayland but adds an extra hop; suggest `xwayland_isolated`).
- X11 host + `xwayland_isolated = true`: refuse (no Wayland to host Xwayland).
- Auto-select: if a template sets `xwayland_isolated = "auto"` or `xephyr_isolated = "auto"`, Project Kennel picks based on host.

## 7.8.6 Clipboard bridging

With `clipboard.bridge = false`, copy-paste between the kennel's X server and the host doesn't work. Users will hit this. Project Kennel's docs are explicit: this is the security boundary, and bridging the clipboard partially defeats it.

For users who turn on bridging, the bridge tool (`autocutsel -fork` or equivalent, run inside the isolated server) becomes part of the kennel's exec allowlist. The diff output shows clearly that clipboard exfiltration is now possible:

```
+ clipboard.bridge = true (direction: bidirectional)
    reason: I want copy-paste from my editor into the AI agent's terminal
    threat impact: WARNING — clipboard is now a bidirectional channel.
                   Context can exfiltrate via clipboard set; can read user's
                   clipboard contents at any time including secrets the user
                   has copied from password managers. Consider one-directional
                   bridging or none.
```

## 7.8.7 Limitations

**Xephyr is not actively maintained.** It still works and ships in every distro's xserver-xephyr package, but the Kdrive codebase hasn't seen significant development in years. Occasional bugs around modern features (touchscreen passthrough, HiDPI, GPU acceleration). Documented as "stable but not improving"; acceptable for the threat model because the surface we care about (X-in-X isolation) has been correct for fifteen years.

**Performance.** Xephyr is software-rendered by default. Terminal applications and IDEs are fine; anything graphics-heavy (browser, GPU-accelerated visualisation) is slow. Project Kennel warns when graphics-heavy applications are detected.

**Xwayland-isolated rendering.** Xwayland renders via Wayland, which is GPU-accelerated. Performance is generally good.

**XKB and locale.** The isolated X server inherits keyboard layout from the host but not always cleanly. The kennel's `setxkbmap` needs to be in the exec allowlist if users will reconfigure layouts.

**Window management.** A bare Xephyr or Xwayland has no window manager. A single fullscreen app works fine; multiple apps need a WM. Templates pick one or leave it to the user.

**The isolated server is still X11.** Inside the isolated server, X11 clients have full X11 capabilities against each other. If the kennel runs a malicious X11 application alongside a trusted one, the malicious one can keylog the trusted one. This is the property Project Kennel is constraining: the *isolation boundary is the X server*, not individual clients within it. Users who run multiple X clients in one kennel should treat them as a single trust unit.

## 7.8.8 Alternatives: VNC, Xvfb, Xpra

For completeness, noting these are not the right tool here:

- **Xnest** is less capable than Xephyr (known issues with composite and render extensions). Use Xephyr.
- **Xpra** is "screen for X11" — runs apps in a detachable X server with remote-display support. Useful for remote dev workflows; its security model is connection-based rather than client-isolation-based. Not a substitute for Xephyr in this design but composes with it (Xephyr inside an Xpra session, for remote kennels).
- **Xvfb** is headless. Useful for build steps that link against X11 but never display, and for CI. A future revision could add an Xvfb option for headless kennels; not urgent.

## 7.8.9 Test plan

For each invariant, a regression test in `tests/x11/`:

1. Policy with `xephyr_isolated = true` and `xwayland_isolated = true`: validator rejects with "at most one X11 path".
2. Policy attempting to grant `/tmp/.X11-unix/X0` in `unix.allow`: validator rejects.
3. Kennel with `xephyr_isolated = true` on Wayland host: Project Kennel warns and suggests Xwayland.
4. Kennel with `xwayland_isolated = true` on X11 host: Project Kennel refuses to start.
5. Context with Xephyr isolated: `xeyes` (or other simple X11 app) runs successfully and displays inside the Xephyr window.
6. Context with Xephyr isolated: attempt to read the host root window via `xwd` from inside the kennel; expect to capture only Xephyr's contents.
7. Kennel with `clipboard.bridge = false`: copy in host, paste in kennel fails (clipboards isolated).
8. Context with Xephyr isolated runs a keylogger script (xinput test); expect it captures only events delivered to Xephyr, not the host.
9. Xephyr `-no-host-grab` flag verified set in process command line.
10. Context exit: Xephyr/Xwayland process terminates and X socket is cleaned up.

Approximately 15 cases in the full corpus.
