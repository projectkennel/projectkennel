# §7.8 Policy surface: per-kennel SSH egress

SSH is a capability domain of its own. A kennel that does real work — an AI
coding agent, a build — needs to `git push`, `git pull`, or `ssh` to a small set
of hosts, using *selected* keys. The naive way to grant that is to expose a
per-kennel `ssh-agent` socket into the kennel (the socket-shim pattern of §7.4).
**That design is unsafe**, and SSH is therefore the one per-kennel service
Project Kennel does *not* hand to the workload as a socket it talks to directly.

## 7.8.1 Why not an agent socket: the destination-blind oracle

The ssh-agent wire protocol carries an opaque to-be-signed blob, **not a
hostname**. An agent — or a fingerprint-filtering broker in front of one — can
therefore bound *which key* signs but never *which host* the signature
authenticates to. A hostile workload that holds the socket opens it directly
(ignoring the stock client and any curated `~/.ssh/config`), requests a
signature for an allowlisted key over a challenge it crafted for an
attacker-controlled host, and authenticates as the user anywhere that key is
accepted — cross-host key reuse. Fingerprint filtering and a curated config
constrain only the client the workload is free to bypass. The agent is a
*destination-blind signing oracle*; see the T1.6 residual in `THREATS.md`.

Per-kennel SSH is routed through a **re-origination bastion** instead. The
workload holds no real key, holds no agent socket, and cannot choose its own
destination.

## 7.8.2 Topology

A single per-user **`kennel-sshd`** — a managed instance of stock OpenSSH
`sshd` — runs alongside `kenneld` for the session, on a loopback port. It holds
no keys; it is a forced-command router whose lifecycle and key state `kenneld`
owns. It is reached over the egress proxy (§7.8.4), so it sits on the network
boundary, not the AF_UNIX boundary of §7.4.

## 7.8.3 The synthetic key is the destination selector

For each `(real-key, host)` edge a kennel is granted (the `[ssh]` policy,
§7.8.8), `kenneld` mints a disposable **synthetic** ed25519 keypair: the private
half goes into the kennel's constructed `~/.ssh`; the public half is bound, via
the bastion's `AuthorizedKeysCommand` (→ `kenneld`), to a forced command that
bakes in the destination and the real-key fingerprint:

```
restrict,pty,command="kennel-ssh-reorigin --dest github.com --key SHA256:<K>" <synthetic-pub>
```

The destination is fixed by *which synthetic key authenticated* — never parsed
from anything the workload sends. A workload holding `synthetic-github` can only
ever reach github with key K; it cannot redirect the forced command, and a
non-synthetic key is refused. There is no oracle: each synthetic key is a
capability for exactly one `(host, key)` edge.

## 7.8.4 Re-origination

On connection, `kennel-ssh-reorigin` (run as the user — no privilege) reads
`$SSH_USER_AUTH` (sshd's `ExposeAuthInfo`) to confirm which synthetic key
authenticated, then `exec`s a fresh `ssh` to the destination using the **real
key from the user's own host-side store** — agent, hardware token, or `~/.ssh`;
Project Kennel stores no key material — `IdentitiesOnly` to the selected
fingerprint, verifying the destination against the bastion's host-side
`known_hosts`. `$SSH_ORIGINAL_COMMAND` is forwarded, so `git`-over-ssh and
interactive shells both work, and the real key signs against a destination the
bastion chose and verified — so cross-host reuse is structurally impossible.

The bastion is reached over the existing per-kennel egress proxy (§7.3): its
loopback port is one allowlisted destination; the kennel's `ssh` targets it and
the proxy forwards. No new transport, no UDS, no helper. Direct `:22` stays
denied by the egress allowlist, so the bastion is the only SSH path out.

## 7.8.5 The synthetic `~/.ssh`

The kennel's constructed `~/.ssh` (a tmpfs subtree of the constructed `$HOME`,
§7.2.5, mode `0700`) holds only the disposable synthetic private key(s); a
generated, read-only `config` — one stanza per granted host, all
`HostName`→bastion, `HostKeyAlias kennel-bastion`, `IdentityFile`→the matching
synthetic key, `IdentitiesOnly yes`, `StrictHostKeyChecking yes`; and a
`known_hosts` carrying only the bastion's host key (under `kennel-bastion`).
Everything real is structurally absent (ENOENT): the user's keys, real `config`,
real `known_hosts`, other kennels' material. The generated config leaks nothing
— it is derived entirely from already-granted policy.

## 7.8.6 Lockdown and scope

`kennel-sshd`'s config and the per-key `restrict,pty` option deny everything but
the forced command and a pty — `AllowTcpForwarding no`, `X11Forwarding no`,
`AllowAgentForwarding no`, `Subsystem sftp /bin/false`. **SFTP, scp, and
port-forwarding are out of scope for the first cut and denied; revisit later.**
Per-signature gating is delegated to the user's key custody: a hardware/sk key
gives touch-per-use for free; a non-interactive key is usable freely *for its
granted destinations only* — there is no Project-Kennel prompt to bypass, so the
`[ssh] allow_headless` flag (§7.8.8) governs whether a non-interactive kennel
may drive such a key at all.

## 7.8.7 Where keys and trust live

- Real private keys live only in the user's host-side store; the bastion uses
  them, the kennel never sees them.
- The bastion's `AuthorizedKeysCommand` helper is **root-owned** (installed in
  the prefix) and queries `kenneld` over its control socket; the rootless
  `kennel-sshd` only invokes it — OpenSSH's safe-path check rejects an AKC that
  is not root-owned.
- The bastion's host-side `known_hosts` = the operator's `known_hosts` ∩ granted
  hosts, or an explicit `[[ssh.known_hosts]]` pin; a granted host with no known
  host key fails closed at `StrictHostKeyChecking`.
- **Hardware tokens** (FIDO/sk keys) are used *host-side* by the bastion, so the
  touch happens on the host. SSH must **not** grant `fido_token` / bind
  `/dev/hidraw*` into the kennel — that would let the workload drive the
  authenticator (CTAP) directly, bypassing the agent and the touch defence
  (§7.7.10).

This is built end to end and validated against stock OpenSSH 9.6. The `[ssh]`
policy schema and its §7.8.8 validators (`kennel-policy`), the synthetic `~/.ssh`
generator and synthetic-key minting (`kenneld::ssh`), the `kennel-ssh-reorigin`
forced-command router, the `kennel-socks-connect` egress connector, the bastion
config + key-state manager (`kenneld::sshd`, `kenneld::bastion`), the
source-only→runtime bridge (`SettledPolicy.ssh`), the `kennel-netproxy`
`[[net.host_services]]` reach, and the spawn-path assembly in `kenneld` are all
implemented. Two end-to-end proofs back it: `src/tools/ssh-bastion-e2e.sh` drives
the real binaries through a bastion + destination `sshd` + agent (re-origination,
command forwarding, the adversarial denies, and the full `ssh` → connector → proxy
→ bastion chain), and the `kenneld` root e2e brings a confined kennel up with an
`[ssh]` grant and verifies its synthetic `~/.ssh` in the sandbox. Production vends
keys through the root-owned `AuthorizedKeysCommand` (`kennel-akc`) that queries the
running `kenneld` for each offered key's forced-command line — the bindings live
only in the daemon's verified edge state, never a file the user could rewrite; a
static `AuthorizedKeysFile` remains the prototype/e2e fallback.
`docs/architecture/08-as-built-notes.md` §8.1 carries the build notes.

## 7.8.8 `[ssh]` policy (source-only)

A compile-time-only section, resolved and folded like `[unix]` and dropped from
the settled `EffectivePolicy`:

```toml
[ssh]
# Whether a granted key may be used by a non-interactive (CI) kennel with no
# per-use touch/confirmation. Loud, threat-tagged; default false.
allow_headless = false

[[ssh.keys]]
fingerprint = "SHA256:…"        # the user's real key, by its stable `ssh-add -l` identity
hosts       = ["github.com"]    # destinations this key may reach (⊆ net.allow on :22)
reason      = "push to the project's github remote"
threats     = { exposed = ["T1.6"] }

[[ssh.known_hosts]]              # optional: pin a host key the operator's store lacks
host = "git.internal"
key  = "ssh-ed25519 AAAA…"
```

Compile-time validation: every `fingerprint` well-formed; every `hosts` entry ⊆
the kennel's `net.allow` on port 22 (otherwise a dead grant or a recon hint);
`allow_headless = true` is loud and carries a threat tag.

## 7.8.9 Test plan additions

Two harnesses realise this plan today. `src/tools/ssh-bastion-e2e.sh` stands up a
real bastion `sshd` + destination `sshd` + agent with stock OpenSSH and drives the
built `kennel-ssh-reorigin` + `kennel-socks-connect` + `kennel-netproxy` through
them, covering items 2, 3, 6 below plus the full `ssh` → connector → proxy → bastion
chain. The `kenneld` root e2e (`src/crates/kenneld/tests/e2e.rs`) brings a confined
kennel up with an `[ssh]` grant and covers item 4 — the synthetic `~/.ssh` (the
generated `config`/`known_hosts` and the synthetic key, nothing real) materialised in
the sandbox. Item 7 is covered by `kennel-policy`'s unit tests (the §7.8.8
validators). Items 1 and 5 (the no-grant route-absence and the two-kennel isolation)
remain as dedicated regression tests.

The invariants, each with a regression test:

1. A kennel with no `[ssh]` grant has no `~/.ssh/agent.sock` and no SSH route: `ssh` to any host fails (direct `:22` denied by the egress allowlist).
2. A kennel granted `github.com` runs `ssh -T git@github.com` / `git ls-remote`; expect the bastion to re-originate with the granted real key. A workload that opens the bastion connection by hand and asks for a *different* destination cannot redirect it — the destination is fixed in the forced command, keyed to the synthetic key it authenticated with.
3. A non-synthetic key offered to the bastion is refused.
4. Read `~/.ssh/`: expect only the synthetic private key(s) and the generated read-only `config`/`known_hosts` — never the user's real keys, real `config`, or real `known_hosts`.
5. Two kennels granted different hosts each reach only their own granted host; a synthetic key minted for one cannot reach the other's host (each is bound by its own forced command).
6. `restrict,pty` + the `Match` block: a port-forward / SFTP / agent-forward request is refused.
7. A `[[ssh.keys]].hosts` entry outside the kennel's `net.allow:22` fails to compile; `allow_headless = true` carries the threat tag.
