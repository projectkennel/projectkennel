A reference monitor for the user level

Confinement, redefined.

Kennel runs code you haven't vetted — an AI agent, an npm install, a freshly-cloned repo — under your own account, confined to just the files, network, and programs a signed policy allows. It's the enforcement the user level never had: the host has boxed untrusted code for decades (SELinux, AppArmor, seccomp), but your account, where agents now run, never did. Kennel keeps your uid and splits the authority off it, so the code runs as you with only what its policy grants, checked one access at a time.

Why

Code under your account is no longer code you wrote.

A modern $HOME runs npm and pip trees, build plugins, language servers, and AI agents that write and execute code for you, pulled in wholesale and mostly unread, all with your keys, your mail, and your source. Under Unix that access isn't a permission the code was granted; it is the code, because it runs as your uid.

The kernel could always contain this (SELinux, AppArmor, Landlock), but those are the OS vendor's to author, so they get switched off rather than used. The boundary that matters moved inside your account; the means to defend it stayed outside. Kennel puts it back: a reference monitor that stays in the path and clears each access, where a sandbox or container draws its line once at launch and steps back.

The dog isn't the threat. It's a good boy that, left loose while you're out, will enthusiastically wreck the house. Not malicious, just eager — and the boundary never has to be right about which kind of dog it turns out to be.

The runtime · Linux

A sandbox with teeth.

Most sandboxes advise; this one enforces. Default-deny throughout: it bounds the files a workload can see, brokers and logs its network, and runs only the binaries you name, on the real kernel and on your actual files. Each capability below traces back to a threat in the catalogue.

01

A granted view, not a copy

The workload sees only the paths you grant: your actual repository, in place. No image to sync, no files shuffled in and out.

02

Allowlisted egress

Outbound traffic runs through a broker that checks it against a list of permitted destinations and records every connection. Anything not on the list simply can't be reached.

03

Only the binaries you name

Execution is limited to the programs the policy permits, so a compromised tool doesn't have the rest of your toolchain to reach for.

04

Hard to scout

Secrets and unrelated trees aren't just unreadable. Where the platform allows it, they aren't visible at all. There's nothing to map.

05

Policy as signed code

Policies are declarative, version-pinned, and signed. A template can't quietly weaken an invariant; resolution refuses it.

06

An audit trail you can keep

Every security-relevant decision is written to an append-only, structured log: meant for a SIEM, not for grepping after something has already gone wrong.

The whole spawn runs unprivileged: kenneld is an ordinary user process holding no standing privilege, and a single small file-capped helper builds the namespaces, dropping to your uid across the clone so the user namespace is operator-owned, not root. There is no sudo anywhere in the spawn. Proven end-to-end on Debian/Ubuntu (AppArmor userns grant) and on Fedora, enforcing SELinux, where a two-domain module keeps the monitor and the workload from ever being the same SELinux subject.

The runtime is Linux, composing Landlock, cgroup BPF, seccomp and namespaces. It isn't the portable part: the threat catalogue and the design are, and a different team could build a runtime to them on another substrate entirely. Other backends (macOS Seatbelt, FreeBSD jails, a microVM) are designed-but-not-built future work, not a current claim.

Install · on Linux

Installed from a repo the project signs itself: never curl | sh.

Kennel installs from signed package repositories the project hosts and signs itself. There is no curl … | sh anywhere in the trust path, on principle: a confinement tool has no business asking you to run unverified code as your user to get it. You import one public key, cross-check its fingerprint against three independent channels — this site, the GitHub release notes, and the domain's DNS TXT record — and thereafter apt/dnf verify every package and every metadata refresh against it. The host holds no key and cannot forge a package.

Debian / Ubuntu

curl -fsSL https://packages.projectkennel.org/kennel-archive-keyring.asc \
  | gpg --dearmor | sudo tee /usr/share/keyrings/kennel.gpg >/dev/null
# cross-check the fingerprint before trusting it:
gpg --show-keys /usr/share/keyrings/kennel.gpg
echo "deb [signed-by=/usr/share/keyrings/kennel.gpg] https://packages.projectkennel.org/deb stable main" \
  | sudo tee /etc/apt/sources.list.d/kennel.list
sudo apt update && sudo apt install kennel

Fedora / RHEL (the .rpm loads the SELinux module for you)

sudo curl -fsSL https://packages.projectkennel.org/rpm/kennel.repo -o /etc/yum.repos.d/kennel.repo
sudo rpm --import https://packages.projectkennel.org/kennel-archive-keyring.asc
sudo dnf install kennel

Signing key 663C 67B0 9FDD A9EE E57F A295 88D5 8446 1C4D 6EE9. Verify out of band: dig +short TXT _kennel-key.projectkennel.org.

then: enable the daemon & run something inside a kennel

# kenneld is an ordinary user process, socket-activated on first use, no root daemon:
systemctl --user enable --now kenneld.socket

# confine an AI coding agent to a repo, a named toolchain, and a few registries:
kennel run claude

# a detachable interactive shell (Ctrl-\ d detaches; reattach later):
kennel run interactive -- /bin/bash
kennel attach interactive

A policy is declarative, signed, version-pinned code: not behaviour, but kernel-level constraint. A leaf policy over the shipped ai-coding-strict template, in part:

# exec: only these binaries may run, nothing else, even if present
[exec]
allow = ["/usr/bin/node", "/usr/bin/git", "/usr/bin/npm", "/usr/bin/python3", …]

# egress: named destinations through the audited broker; all else denied
[[net.proxy.allow]]
name = "registry.npmjs.org"
ports = [443]
threats.exposed = ["T1.9"]   # each grant traces to a threat in the catalogue

# filesystem, identity, audit, ssh … inherited from base-confined@v1, then narrowed

The CLI · two houses

Run touches settled artefacts; authoring owns source, templates, and keys.

One binary, two contexts: run and its neighbours only ever start or manage a signed, settled kennel, while the policy / template / key houses author and sign the things you run. You can't hold the wrong object at the wrong verb.

kennel run <policy> [-- <cmd>] run · start a confined workload; an interactive run detaches with Ctrl-\ d kennel attach / stop / list run · reattach a terminal; stop a kennel; list kennels + the service mesh kennel oci run <policy> run · boot a digest-pinned OCI image as the kennel root kennel review <path> run · re-pin a workspace trust manifest, or --revert tampering kennel policy compile / install author · resolve + sign a leaf into a settled, byte-pinned artefact kennel policy validate / risks / diff author · check a policy; score it against the threat catalogue; show grant deltas kennel template sign / lint author · sign a shared base template or fragment; check the corpus kennel key generate / rotate / trust author · manage tier-bound signing keys (user; the host key as root)

Built and shipping, versus what's still on the roadmap.

Runs today

  • The full unprivileged spawn vertical (~3.7 ms construction, non-host modes)
  • Signed apt/dnf repos — Debian/Ubuntu + enforcing-SELinux Fedora
  • Four net modes · brokered egress proxy + cgroup-BPF · D-Bus facade
  • Dynamic spawn + cross-kennel service mesh, over binder IPC
  • Confined GUI (nested Wayland compositor) · OCI image substrate
  • SSH re-origination bastion · AF_UNIX shim · workspace-trust pinning
  • Signed templates & fragments · policy compiler · lockfile · unified audit

Designed, not yet built

  • A macOS port (Seatbelt / Mach ports — the mapping exists, the runtime doesn't)
  • Cross-instance binder relay (richer cross-kennel MCP topology)
  • Per-method policy on mesh grants (interface / method granularity)
  • An OCI integrity ladder above the digest-pinned floor

Declined on principle (not deferred): a secrets or signing broker, a first-party OCI unpacker, and daemon-side protocol mediation — each would put a trust root or an untrusted parser where it doesn't belong. The reasoning is in the book.

How

Not a new sandbox: a careful arrangement of what your OS already ships.

Kennel doesn't reimplement the kernel or sit in front of it. It composes the enforcement the platform already provides — Landlock, cgroup BPF, seccomp, namespaces — and wires it to one signed policy, everything starting from deny. A small file-capped helper does the few steps that genuinely need root; the workload never does.

Because it governs what a workload can reach rather than the host kernel's attack surface, it sits under stronger isolation when a job needs it: run a kennel inside a gVisor sandbox or a microVM to add host-kernel containment without losing the egress control, recon-resistance, or audit trail. Different layers, stacked only when it's worth it.

Between two options that don't quite work.

Light enough to leave on, measured

Cheap enough to be disposable.

Time to stand up a fresh, isolated execution context. Kennel builds confinement per task, not per session, so the cost has to be small enough to pay on every tool call, not just every login.

Project Kennel ≈ 3.7 ms Firecracker microVM ~ 275 ms Apple container ~ 600 ms

Construction measures ≈ 3.7 ms (median, timed to the workload running, spawn-spinup.sh): roughly 75× under a Firecracker microVM and 150× under Apple's container, all measured the same way. The boundaries differ, though: a microVM isolates the box (one tenant from the next) but says nothing about what code does inside it; Kennel governs what each workload reaches. So they compose — a microVM underneath for the hardware boundary, Kennel inside to bound reach. Faster still is Microsoft Hyperlight at ~0.9 ms, but that embeds a single function with no OS: a different job.

The nearest neighbour

"Isn't this just bubblewrap?"

Kennel builds its sandbox with the mechanism bubblewrap pioneered; the difference is reference-monitor discipline. bwrap composes a view and execs: it mediates once, at launch, then steps out. Kennel mediates at every crossing, at runtime, through kenneld. Same engine, different vehicle:

Same engine

  • Unprivileged, identity-mapped user namespace: CAP_SYS_ADMIN only inside it, the workload never root on the host
  • Mount / PID / IPC / (opt.) net namespaces; pivot_root
  • A minimal bound view: present by construction, not denied
  • seccomp on the syscall surface

Different vehicle

  • A signed, declarative, versioned policy (templates, inheritance, a trust chain), not imperative CLI flags
  • Landlock + cgroup-BPF egress allowlist + a brokering SOCKS proxy + the SSH re-origination bastion + a D-Bus facade, atop namespaces + seccomp
  • A daemon (kenneld): lifecycle, attach/detach, supervision, the service mesh, dynamic spawn
  • A unified audit trail, and a threat catalogue the policy answers to

The gap is real: a Claude Code agent blocked by a bubblewrap sandbox reasoned, unprompted, "let me try disabling the sandbox … to bypass the deny rule" (THREATS.md T2.1); a launch-time view has no runtime mediator to catch that. bubblewrap is the right primitive — it's what Flatpak runs on — and Kennel is the policy-driven, audited, daemon-managed reference monitor built from it: one signed, reviewable policy instead of a wall of bwrap flags.
